# Render depth + the backgrounded-outcome join

**Read `docs/gotchas.md` first** — entry 5 (the degradation law) is what
this issue makes visible.

The evidence the campaign added becomes readable, and one facet gets deeper:
the backgrounded-command outcome that v1 declared ABSENT-final is now
resolvable (the recording pass proved the join exists), while the unpolled
tail stays honestly ABSENT.

## Render (R12)

- **log** gains two columns on the existing one-line-per-session format
  (additive — the PRD-0001 pinned minimum fields all remain): a
  **checks** column summarizing the session's bound checks (pass/fail
  derived from exit code 0), and a **cost** column carrying the derived
  marker. Standalone checks (session NULL) belong to no session line and do
  not render in `log` in v1 — they remain reachable in the ledger; name
  this in a comment, do not silently drop them.
- **show** heads the timeline with the cost line (derived marker), keeping
  the existing sha/footprint header intact, and renders badge lines: one
  per bound check (name, pass/fail, truncated flag when set) and one per
  test-results facet (counts, failed names, duration).
- **Absent is explicit everywhere**, through the one existing absent-marker
  module: cost-absent and test-results-absent are the asserted cases. A
  claimed test run with zero tests renders its real zeros — never the
  absent marker (row membership is the distinction; the readers already
  encode it).
- The **derived marker** distinguishes computed evidence (cost, from the
  pinned price table) from observed evidence; it is one shared rendering
  helper, not per-call-site string literals.

## The backgrounded-outcome join (R14)

v1's command outcome is a derived three-state: success / failure / ABSENT
when backgrounded. The third state now resolves — as pure derivation at
show time, with no stored outcome column and no spool mutation:

- The backgrounding `PostToolUse` carries the task id at
  `tool_response.backgroundTaskId`; a later `PostToolUse` of the TaskOutput
  tool carries `tool_input.task_id` and
  `tool_response.task.{status, output, exitCode}`.
- Ingest promotes BOTH payload locations, normalized, into the events
  table's nullable `background_task_id` join-key column (the column exists;
  populating it is this issue's ingest touch). NULL on every other event.
- The outcome derivation joins within the session, in memory over the
  already-loaded events: matched TaskOutput with exitCode 0 → success;
  matched with nonzero → failure; **no matching TaskOutput (no poll before
  session end, or a crash) → ABSENT, honestly** — no poll, no join, no
  guess. Both surfaces are register entries (undocumented tool + response
  shape); any shape surprise degrades to ABSENT, never throws.

## Invariants (contract prose, not acceptance)

- Every PRD-0001 render assertion stays green: the additions are new
  columns, header lines and badges — existing fields, their order within a
  line, and the existing absent-marker cases are untouched.
- No new index and no materialized outcome: the join is per-session and
  in-memory; the ledger accretes no state the spool cannot rebuild.
- The spool and the hook artifact are untouched.

## Test-harness contract

Copy `tests/acceptance/harness/` verbatim; acceptance at
`tests/acceptance/ISS-0024/` drives the built CLI as a subprocess with
fixture replay.

Specifics the test-author must not go discover:

- The background scenario loads by name through the typed loader (16
  events, recorded on 2.1.211); its expected join facts (task id, exitCode
  0) are pinned in the transcripts manifest's probe metadata — read them,
  do not grep the stream by hand.
- **The truncated variant is derived in the test, in memory**: replay the
  loaded background stream cut off before the TaskOutput event that carries
  the completed poll. Nothing hand-authored is committed; the manifest is
  untouched.
- Cost-present rendering replays with the fixtures layer's
  transcript-substituting replay wrapper (fixtures issue); cost-absent
  replays without it. Checks
  come from running the real check command against an open session (the
  SIGKILL scenario is the open-session recipe — see the check issue's
  tests for the pattern; here it is enough to run one passing and one
  failing check bound to the session).
- test-results-absent is any session whose stream has no vitest run (the
  headless scenario); test-results-present replays the vitest scenario.
- Column/badge assertions parse the command's actual stdout (the seam),
  not renderer internals; unit tests under `tests/unit/render/` and
  `tests/unit/facets/` cover formatting helpers and the join derivation
  (matched-zero, matched-nonzero, unmatched, hostile task shapes → ABSENT)
  as pure logic.
