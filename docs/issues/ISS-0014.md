# Absence-reasons record — the drift detector's contract

**Read `docs/gotchas.md` first**, especially entry 5 (the degradation law):
this contract IS that law made queryable.

The product's thesis applied to itself: whenever ingest degrades a facet to
ABSENT, the WHY must be recorded — never silently. This issue ships the
typed record and its write/read API over the `absences` table that ISS-0013
already created (`PRIMARY KEY (session_id, facet)`). Three later slices
consume it: cost enrichment writes cost absences, the drift detector writes
kind absences, and doctor reads the whole set. The detector logic itself is
NOT here — only the contract those slices share.

## The record

`(session_id, facet, reason)`, grain per-session × facet — one expected slot
per facet per session, which is exactly what doctor enumerates. It is a
derived projection: recomputed on each ingest, rebuildable from the spool
plus transcripts-at-path, holding zero ground truth of its own.

**The vocabulary is closed** (the fragile-dependency register in
`docs/spec-v1.md` is the enumeration; these exact strings, no synonyms):

| facet | reasons |
|---|---|
| `cost` | `transcript unavailable` · `transcript shape unrecognized` · `model unpinned: <model>` |
| `kind` | `no SessionStart captured` · `model absent, contradicted by end-reason` |

Export the facet names and reason builders as typed constants (the
`model unpinned` reason is a builder taking the model string). The types
make an out-of-vocabulary facet unrepresentable; reasons for a facet are
constrained to that facet's set.

**Deliberately excluded facets** (their ABSENT is already structural, and
duplicating it here would be redundant or flooding): command outcome (a
payload signature / the backgrounded join), test-results (row membership —
a non-test command is not a degraded facet and must not emit an absence per
command), and the nullable sha/worktree columns (their NULL is
self-describing).

## API shape (design considered twice)

Chosen: **recompute-upsert semantics** — a per-session "set the absences for
facet F" writer with `INSERT ... ON CONFLICT(session_id, facet) DO UPDATE`,
plus an explicit clear when the facet is present, so each ingest's derived
recompute converges to the current truth and is idempotent (re-run changes
no row counts). Readers: absences for one session; absences across all
sessions (doctor's query, served by the primary key).

*Rejected shape, recorded:* an append-only absence event log (spool-style),
folded on read. Superficially consistent with the house pattern, but wrong
layer: the spool is already the ground truth these rows derive from, so an
intermediate log would be a second derived store to keep consistent, and
"current absences" would need fold logic everywhere the table is read. The
ledger row IS the fold; delete-and-reingest is the recovery story.

## Invariants (contract prose, not acceptance)

- An absence row never appears for a facet that is present — presence and
  absence-with-reason are mutually exclusive by construction.
- The table never fabricates: no caller may write an empty-string reason or
  a reason outside the facet's enumerated set.
- Rebuild-equivalence is inherited, not implemented here: because writers
  recompute from spool + transcript, delete-ledger + re-ingest converges to
  the same rows (asserted at the seam by the consuming issues).

## Test-harness contract

Unit tests only, against a real ledger file in a tmpdir (open it through the
core ledger module — do not hand-create SQLite schema). No fixtures, no
subprocesses, no seam. Cover: upsert idempotency (row counts stable across
re-runs), reason replacement, clear-on-recovery, the closed vocabulary
(a facet outside the enumeration must not typecheck / must be rejected), and
both readers. The consuming issues own the end-to-end proof that real ingest
emits these rows.
