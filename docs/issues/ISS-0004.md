# The hook artifact: capture and boundary enrichment

The hook artifact is the one piece of coreartifact that runs **inside foreign
sessions**, on the host's hot path, in repos it does not control. It is `high`
risk for that reason alone: a bug here degrades or breaks somebody's Claude
Code session. Its contract is deliberately tiny.

## Contract

- **Self-contained and zero-dependency.** It must run in a repo with **no
  `node_modules`** — the target repo's, and possibly none at all if the
  package was invoked via `npx` and its cache was later evicted. `init` copies
  the artifact *into the target repo* and the hook config references it by
  **absolute path**. Therefore the artifact is a single file that imports
  nothing but `node:` builtins. No imports from `src/core`, no bundler, no
  transitive anything.
- **Always exits 0.** Every failure mode — unparseable stdin, unwritable
  spool, git shell-out failure, a missing directory — is swallowed and the
  process exits 0. Capture is best-effort; the host session is sacred. A
  failure must also never leave a partial line in the spool: build the whole
  line in memory and write it with **one** atomic `O_APPEND`.
- **Exactly two behaviors**, and nothing else: the append and boundary git
  enrichment (2026-07-14 amendment — WorktreeCreate propagation removed, see
  below). It does not read the ledger, does not read transcripts, does not
  touch the network, does not print to stdout.

The zero-import constraint means the artifact re-states the envelope shape
rather than importing it. That duplication is accepted and must be **pinned**:
a unit test asserts that a line the artifact writes parses cleanly through the
core envelope parser. If the envelope contract ever changes, that test goes red
in both places at once, which is the point.

## Behavior 1 — the append

Read the hook payload from stdin. Write exactly one spool line:

```
{ "v": 1, "ts": "<iso8601>", "event": <hook payload, verbatim> }
```

The payload is **byte-preserved**: the stdin bytes are embedded as the `event`
member without re-serialization. Do not parse-then-stringify — key order would
drift and the "payload stored verbatim, never rewritten" invariant would be
silently violated. (The artifact may still *read* fields out of the payload to
decide whether it is a boundary or a WorktreeCreate event; that read must not
feed the write.)

The spool is the per-repo append-only JSONL file under
`<repo_root>/.coreartifact/`. Which repo root? **Attribution**: resolve the
cwd's git common dir to the main repo root, so a session running in a git
worktree appends to the **main checkout's** spool. A non-git cwd falls back to
the init root, which `init` bakes into the installed hook config as an
argument or environment value — the artifact must not have to guess it.

Concurrency is real: several sessions and worktrees append to one spool at
once. A single `O_APPEND` write of one line under the pipe-buffer size is
atomic on macOS and Linux; that atomicity is the whole concurrency design.
Do not read-modify-write the spool. Do not lock it.

## Behavior 2 — boundary enrichment

On SessionStart and SessionEnd only, add a top-level `git` sibling of `event`:

```
"git": { "head": "<sha>", "dirty": <bool> }
```

Both follow the degradation law: **present with a value, or ABSENT (key
omitted)** when git resolution fails — a non-git cwd, or a repo with no
commits. Never an empty string, never a fabricated `false`. The dirty flag
comes from whether the working tree has changes at boundary time.

No other event is enriched. This is the whole of the hook's intelligence: the
thin envelope plus boundary shas, nothing more. Cost, tokens and transcript
content are explicitly not this artifact's business.

## No worktree behaviors (2026-07-14 amendment — the escalation fired)

The recording pass resolved the WorktreeCreate open risk in the worse
direction (see `docs/recording-pass.md`): **WorktreeCreate is a delegation
hook** on Claude Code 2.1.209 — a configured hook must create the worktree
and print its path to stdout; a passive append-only hook fails the spawn
("hook succeeded but returned no worktree path"), breaking every
`isolation: worktree` agent in the repo. The payload carries no worktree
path (only a `name`), and WorktreeRemove never fired at cleanup.

Therefore the artifact has **no** WorktreeCreate or WorktreeRemove behavior,
and init subscribes neither event. Do not add them back "for completeness" —
a passive subscription is host-breaking, and a delegating one would make
coreartifact hijack worktree creation, which capture must never do.
Propagation is init-time copy plus the ingest gap warning; worktree-isolated
subagents are captured through the parent session's hooks regardless
(observed). A hand-made `git worktree add` with no session involved stays
uncaptured until the next ingest warning names it. That residual is accepted.

## Invariants (contract prose, not acceptance)

- Nothing leaves the machine. The artifact makes no network call, ever.
- Transcripts are referenced by the path inside the payload, never read, never
  copied.
- The spool is append-only: the artifact never rewrites, truncates, rotates or
  compacts it.

## Test-harness contract

Copy the `tests/acceptance/` harness from the harness issue **verbatim** — the
tmpdir-repo factory, the CLI runner, the fixture replayer and the worktree
helper. Do not build a second way to spawn processes or make repos.

Acceptance tests at `tests/acceptance/hook.test.ts` drive the seam: create a
tmpdir git repo, **install the artifact the way `init` will** — copy the built
artifact into the repo's `.coreartifact/` directory and invoke it by absolute
path with the init root supplied exactly as the installed hook config will
supply it — then replay fixture streams through it with the replayer and
assert on the **spool bytes**: line count, envelope validity, byte-preserved
payloads, boundary `git` members. This issue must **not** call `init` (the
init slice depends on this one, not the reverse); the harness's fixture
replayer takes the hook command as an argument precisely so the artifact can
be exercised standalone. The parallel-replay case uses the replayer's
interleaved mode: N streams at once, and the spool line count must equal the
sum of the inputs. (The plan-gate propagation criterion that briefly lived
here was removed by the 2026-07-14 amendment along with the behavior itself.)

The exact contract by which the artifact learns its init-root fallback (an
argv argument or an environment value baked into the installed hook config) is
**owned by this issue** and consumed by init. Name it explicitly in the module
so the init slice wires it rather than re-deriving it.

Fixture streams come from the fixtures issue's loader by scenario name
(`headless`, `interactive`, `worktree`, `SIGTERM`, `SIGKILL`). Do not
hand-author payloads here; a payload shape this issue needs and the fixtures
lack is a routed change to the fixtures issue, never a drive-by.

Below-the-seam unit tests under `tests/unit/hook/` cover only the artifact's
pure parts (boundary detection, envelope framing) and the cross-check that a
written line parses through the core envelope parser.

The no-node_modules test must genuinely have none: build the artifact, copy it
to a tmpdir repo that contains no `node_modules` directory, and run it there by
absolute path.
