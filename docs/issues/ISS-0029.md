# Session endpoint + freshness, concurrency, zero-write

`GET /api/session/<id>` — the same evidence `show` derives, structured as JSON
(api.md Surface D). Also the home of R5 (freshness + concurrency) and R8
(zero-footprint reads), because by this issue both endpoints exist and both can
be exercised in one browse. Binding shape, field names, and the null-vs-missing
rule are api.md — fixed.

## Contract

`src/dashboard/session.ts` is the handler wired into `src/dashboard/routes.ts`
(replacing the ISS-0027 stub). Resolution reuses the ingest-on-read + union
path `resolveSession` (`src/resolve-session.ts`): it ingests each reachable
repo, then reads the matched ledger's rows for `<id>`. The response mirrors
exactly what `show` already derives — reuse the derivation, never re-invent it:

- **facets** — the header. `status` is never null. `kind` is
  `headless`/`interactive`/`null` (a null kind carries its `absences` reason,
  facet `kind`). `sha_before`/`sha_after`/`ended_at` are self-describing
  nullables (no `absences` row). `worktree_path` is `string|null`,
  self-describing (`null` = the main checkout — PRD Amendment 1, a first-class
  session column the viewer must not hide). `cost` = `{ value, derived: true }`;
  `tokens` = `{ derived: true, input, output, cache_read, cache_creation }`
  (all-present or all-null together). `model`/`cc_version` are bare
  `string|null`.
- **checks** — one entry per **bound** check (`checks WHERE session_id = ?`):
  `name`, `exit_code` (carried — a 137/SIGKILL reads differently from a 1),
  `passed` (the derived render of `exit_code == 0`), `truncated`, `bound_by`.
  Standalone checks never appear.
- **test_results** — one entry per parser-claimed command; **empty array =
  facet ABSENT** at the session level (no command was claimed), distinct from a
  present row with `passed=failed=skipped=0` (a real zero-test run — B1 rule 4).
  `duration_ms` is `int|null`; `failed_names` is a real array (`[]` = known
  empty, never null).
- **footprint** — `string[]` of distinct touched paths; `[]` is a real empty
  set, not ABSENT.
- **absences** — `{ facet, reason }` verbatim from the `absences` table
  (`facet ∈ {cost, kind}`); the collection every explained `null` points to.
- **timeline** — flat, `seq` order, **uncapped** (api.md Surface D: a single
  session's event count is bounded; a cap would misorder the evidence). Each
  entry carries the derived facets `show` already computes plus the four
  nesting keys passed through verbatim (`prompt_id`, `agent_id`, `agent_type`,
  `tool_use_id`, present on every entry, `null` where absent). Entry kinds:
  `lifecycle` (`hook_event_name`), `prompt` (`""` = real empty prompt, not
  ABSENT), `command` (`command`, `duration_ms`, `outcome`, `test_results`),
  `subagent`. `outcome` is the **three-state facet resolved server-side exactly
  as `show` resolves it** (`src/facets/outcome.ts`, including the
  backgrounded-outcome same-session join over `background_task_id`):
  `{state:"success"}` | `{state:"failure", error}` | `{state:"absent"}`. Never
  collapsed to two states; a backgrounded command with no resolving `TaskOutput`
  stays `absent`, honestly. Raw event payloads are NOT carried — the keys and
  derived facets are the interface (the dashboard is a viewer, not a spool
  browser; the tree view is a v2 non-goal — surface the keys, interpret
  nothing).

### Errors (api.md Surface A + Flag 2)

Unknown id → `404 unknown_session` naming the id. A **bare** id that resolves
in more than one repo (`resolveSession`'s `ambiguous` case — the same fixture
replayed into two repos) → `404 unknown_session` stating the id does not
*uniquely* resolve and naming the candidate roots. No fourth status; the UI's
own links always carry `?repo=<root>` (which scopes resolution to one root) so
the UI never hits ambiguity.

### Concurrency (R5)

The session endpoint's own read connection to the matched ledger must set
`PRAGMA busy_timeout = READ_BUSY_TIMEOUT_MS` (import the constant from
`src/dashboard/constants.ts`) before reading, so a concurrent writer never
surfaces "database is locked." The union walk it goes through
(`walkRegisteredRepos`) already sets the pragma via the ISS-0028 amendment this
issue depends on.

## Test-harness contract

Copy `tests/acceptance/harness/` verbatim. Seed sessions by replaying fixture
streams through the hook (kind mapping and real-`check` binding exactly as in
ISS-0028's contract — reuse that seeding). Assert raw JSON at the HTTP seam.

- **Facet completeness / ABSENT.** Replay a session with a present transcript
  (via `buildSubstitutedTranscript`, `tests/fixtures/transcriptReplay.ts`) for
  the derived cost/tokens, and a `clear-source` (kind-ABSENT) session for the
  null-kind + `absences` assertion. Assert `null` (not `0`, not omitted) for
  every ABSENT facet.
- **Ambiguity.** Register two tmpdir repos and replay the *same* fixture stream
  into both (identical `session_id` in two ledgers), then GET the bare id and
  assert the descriptive `404 unknown_session`.
- **Freshness (R5).** With the server running (spawned `open --port 0
  --no-browser`), append a valid line to a registered spool and assert the very
  next overview GET and the very next session GET reflect it — no restart, no
  poll. Because this drives an overview GET too, this issue depends on ISS-0028.
- **Concurrency (R5).** Use a **separate process** to hold the ledger during
  ingest while a GET runs (gotcha #4: a `Promise.all` over a synchronous body
  runs serially and cannot detect the lock it exists to catch — spawn a real
  concurrent writer). Assert both complete with no "database is locked" surface,
  for both endpoints.
- **Zero-write (R8).** Snapshot the repo tree outside `.coreartifact/` and the
  registry log, browse both views over HTTP, and assert byte-identity and that
  the registry has not grown (the only sanctioned write is the ingest
  projection inside `.coreartifact/`).

## Invariants (reviewer prose, not acceptance)

No schema change. The server never reads a transcript beyond ingest, never
executes `claude`, never mutates the spool, performs no action. `null`-vs-
missing per api.md B1 is uniform. `show`'s own CLI output is unchanged.
