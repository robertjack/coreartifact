# Core contracts II: the ledger and the registry

Split out of ISS-0001 on 2026-07-14 after it escalated twice. This issue is
**persistence**. Its sibling ISS-0011 is attribution. Between them they hold
every part of the original contracts issue that actually failed.

Prior art: `escalated/ISS-0001-attempt-1` and `escalated/ISS-0001-attempt-2`
contain working ledger and registry implementations. Reuse the ledger nearly as
is. **Do not reuse the registry's locking code** — it is the thing being
deleted (see below).

## The ledger (SQLite, schema v1)

At `<repo_root>/.coreartifact/ledger.db`, alongside the spool. A **disposable,
rebuildable projection** — deleting it and re-ingesting the spool must rebuild
equivalent rows. The DDL is fixed (see `schema.md`, which is normative):

```sql
CREATE TABLE meta (
  id             INTEGER PRIMARY KEY CHECK (id = 1),
  schema_version INTEGER NOT NULL,
  ingested_bytes INTEGER NOT NULL DEFAULT 0,
  lines_seen     INTEGER NOT NULL DEFAULT 0,
  last_ingest_at TEXT
);

CREATE TABLE sessions (
  session_id     TEXT PRIMARY KEY NOT NULL,
  repo_root      TEXT NOT NULL,
  worktree_path  TEXT,
  kind           TEXT CHECK (kind IN ('headless','interactive')),
  status         TEXT NOT NULL
                 CHECK (status IN ('open','closed-clean','closed-inferred')),
  sha_before     TEXT,
  sha_after      TEXT,
  started_at     TEXT NOT NULL,
  last_event_at  TEXT NOT NULL,
  ended_at       TEXT
);

CREATE TABLE events (
  line_no         INTEGER PRIMARY KEY,
  session_id      TEXT NOT NULL,
  seq             INTEGER NOT NULL,
  ts              TEXT NOT NULL,
  hook_event_name TEXT NOT NULL,
  prompt_id       TEXT,
  agent_id        TEXT,
  agent_type      TEXT,
  tool_use_id     TEXT,
  payload         TEXT NOT NULL
);

CREATE TABLE footprint (
  session_id TEXT NOT NULL,
  path       TEXT NOT NULL,
  PRIMARY KEY (session_id, path)
);

CREATE INDEX idx_events_session ON events (session_id, seq);
```

No other tables, no other indexes. `openLedger`'s contract is **open/create**:
it creates the parent `.coreartifact/` directory when absent rather than
throwing SQLite's opaque "unable to open database file".

**NULL means ABSENT** — the degradation law encoded structurally. The `CHECK`
constraints on `status` and `kind` exist to make fabrication a schema violation.

If `node:sqlite` is the driver, `engines.node` must match it (unflagged only
from Node 22.13); `package.json` and `pnpm-lock.yaml` move together.

## The registry — append-only, no locks (2026-07-14 rewrite)

**Read `schema.md`'s "Surface 3" section; it is normative.** The short version,
and the reason this issue exists:

The registry used to be a JSON object holding an array, so `addLedger` had to
read-modify-write. That is the **only** read-modify-write in the system, and it
was the only place with concurrency bugs. Three consecutive review rounds died
on it: a lost update, then an O_EXCL lock that permanently wedged every `init`
on the machine after one crash, then a stale-lock steal that was TOCTOU and lost
updates again. Each fix created the next bug.

So the read-modify-write is **deleted**, not fixed. The registry is now an
**append-only JSONL log** at `~/.coreartifact/registry.jsonl` — the same shape
as the spool, which is the pattern this whole product is built on:

```
{"v":1,"op":"add","repo_root":"/abs/path","at":"<iso8601>"}
```

- **`addLedger` is one atomic `O_APPEND` of one line.** It does not read the
  file. There is **no lock file**, no reaping, no TTL, no pid check, no
  stale-lock steal, no retry loop. If you find yourself writing any of those,
  you have taken a wrong turn — the acceptance criterion asserts no lock file is
  ever created. Concurrent `init` runs across parallel worktrees cannot lose
  each other's entries: a single append of one line under the pipe-buffer size
  is atomic on macOS and Linux, exactly as it is for the spool.
- **`readRegistry` folds the log into a set**, deduping by `repo_root` (last op
  for a root wins). It is **total**: a corrupt or truncated line is skipped and
  counted, never thrown. A missing file folds to the empty set. A damaged
  registry must never take down every command that reads it.
- **Idempotence is a property of the fold, not the write.** Re-running `init`
  appends a second `add` line for the same root; the fold still yields one
  entry. The file grows only on repeated `init` — rare — and compaction is a
  v1.1 concern.
- **`op`** exists so PRD-0002's uninstall can append a `remove` tombstone rather
  than rewriting. Append-only stays append-only.

The uniform architecture this buys: **append-only logs are ground truth
everywhere; every readable structure is a fold over one.** Spool → ledger.
Registry log → registry set.

## Invariants (contract prose, not acceptance)

- The registry and spool are append-only: never rewritten, never truncated,
  never rotated, never compacted (v1).
- The ledger is disposable; the spool is truth.
- Nothing leaves the machine.
- `package.json` keeps `private: true`.

## Test-harness contract

Unit tests only (plain vitest) under `tests/unit/core/`. Create ledgers and
registries in the test's own tmpdir — **never** touch a real `~/.coreartifact`
(use the registry-root environment override that ISS-0001's paths module
exposes).

The concurrency criterion must be a **real** concurrency test: N genuinely
concurrent `addLedger` calls (Promise.all over N appends, or N spawned
processes), then assert the fold contains all N roots. A test that serializes
the calls proves nothing about the property being claimed.

An acceptance harness exists at HEAD (`tests/acceptance/`, from ISS-0002); this
issue does not add to it. `pnpm test` must be green when you finish.
