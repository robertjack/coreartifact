# The acceptance harness: tmpdir-repo factory, CLI runner, fixture replayer

**Prior art: none — this is the first campaign.** This issue *creates* the
harness that every subsequent slice copies. Every later issue's Test-harness
contract says: copy the `tests/acceptance/` harness from this issue verbatim.
Do not re-derive it, do not fork it, do not invent a second way to spawn the
CLI.

## The seam (one, and only one)

Acceptance tests build the CLI and **drive it as a subprocess** in fresh
tmpdir git repos (worktrees included), replay recorded payload fixtures by
piping them into the installed hook command exactly as Claude Code would
deliver them, and assert on stdout, exit codes, spool bytes and ledger rows.

Real filesystem, real git, real SQLite, **zero mocks at this seam**. If a test
reaches for a mock, it is either testing pure logic (which belongs in a unit
test below the seam) or it is testing the wrong thing.

## What the harness provides

Four primitives, exported from one place:

1. **tmpdir repo factory** — creates a fresh temporary directory, `git init`s
   it, sets a deterministic user identity, makes an initial commit, and hands
   back the repo root plus an **isolated HOME** and an **isolated registry
   path**. Cleanup removes the whole directory.

   Isolation is the load-bearing part. The registry is a *global* file at
   `~/.coreartifact/registry.jsonl`; an acceptance test must never read or write the
   operator's real one. The CLI's paths module exposes an environment override
   for the registry root (defined in the core contracts issue) — the harness
   sets it, and every subprocess it spawns inherits it. A test that leaks into
   the real home is a defect in the harness, not the test.

2. **CLI runner** — builds the CLI once per test run (the build is a
   pre-step, not per-test) and invokes the built entry as a subprocess with a
   given cwd, argv, HOME and registry override, returning exit code, stdout
   and stderr. Invoke the built entry by path via `node`; do **not** depend on
   installed bin names — bins are the packaging slice's concern and its own
   acceptance test covers them.

3. **fixture replayer** — loads a fixture stream by scenario name (through the
   fixtures issue's loader — never a raw path) and pipes each payload line into
   a supplied hook command on stdin, one process invocation per line, in order,
   delivering the payload bytes unchanged. Returns each invocation's exit code.
   It must also support **N interleaved parallel replays** into the same hook
   command, since the capture slice must prove that concurrent sessions lose
   zero lines.

4. **worktree helper** — `git worktree add` against a tmpdir repo, returning
   the new checkout path.

## Design constraints

- **Deterministic and hermetic — via an ALLOWLIST, not a denylist**
  (2026-07-14 finding; the codebase already ruled on this in
  `src/core/attribution.ts`). Every git and CLI subprocess the harness spawns
  gets an environment **built from scratch** — `PATH`, `HOME`, `XDG_CONFIG_HOME`,
  and the explicit `GIT_AUTHOR_*` / `GIT_COMMITTER_*` — never `{...process.env}`
  with a few names deleted. Deleting `GIT_DIR`/`GIT_WORK_TREE` while leaking the
  operator's `GIT_CONFIG_GLOBAL`, `GIT_COMMON_DIR`, `XDG_CONFIG_HOME` etc. is
  exactly the denylist ISS-0011 forbade — and here it would make a "hermetic"
  tmpdir repo silently depend on the operator's machine, poisoning every
  downstream slice's tests. Reuse ISS-0011's allowlist rather than inventing a
  second one; the CLI runner must pass the same clean env to the CLI subprocess.
- **The build is once per RUN, not once per worker fork** (2026-07-14 finding).
  Vitest's default `forks` pool gives each test *file* its own process, so a
  module-level `buildPromise` memo builds **N concurrent `tsc` processes all
  writing the same `dist/`** — a race that corrupts the shared output. Build the
  CLI **before vitest starts** (a `globalSetup`, or a `pretest` step whose
  output the runner asserts is present), and have the runner *verify* `dist/` is
  fresh rather than *trigger* the build. A cross-worker build cannot be
  memoized inside a worker.
- **Freshness must cover the whole build input**, not just `src/`: a
  `tsconfig.json` change, or a restored-older source file, must invalidate a
  stale `dist/`. Watch the real inputs (or rebuild unconditionally in the
  pre-step, which is simplest and cheapest here).
- **Assertion helpers over raw fs.** Provide small readers: read the spool as
  a list of parsed envelope lines; open the ledger read-only and query
  sessions / events / footprint rows. Later slices assert on ledger rows
  constantly; each re-implementing SQLite access is how drift starts.
- **Cheap.** These tests spawn processes; keep the per-test cost to a repo
  init plus the invocations the test actually needs. Share the build, never
  the tmpdir.

## Test-harness contract

The harness's own self-test lives at `tests/acceptance/harness.test.ts` and
exercises **all four** primitives against the real filesystem — including the
parallel replayer, which shipped **untested** in the first attempt even though
the capture slice (ISS-0004) depends on it to prove concurrent sessions lose
zero lines (2026-07-14 finding). The self-test must cover: the factory creates
and cleans up a real git repo; the runner spawns the real CLI (usage, exit 0 on
no args) **with a hermetic allowlisted env**; the serial replayer pipes a real
fixture stream into a stub command that records the stdin it received; **the
PARALLEL replayer interleaves N streams into one stub and the stub receives
exactly the sum of the input lines, none lost**; and the worktree helper
produces a real worktree whose git common dir resolves to the main repo.

Prove hermeticity explicitly: a self-test that sets a hostile `GIT_CONFIG_GLOBAL`
/ `XDG_CONFIG_HOME` / `GIT_COMMON_DIR` in the parent env and asserts a
harness-made repo is unaffected. Standing rule: *could this test fail if the bug
were present?* The env one must fail if the allowlist regresses to a denylist.

Fixture streams come from the fixtures issue's loader by scenario name.
