# Packaging: the tarball, the two bins, and the npx path

**Before writing anything, read `docs/gotchas.md`** — the bugs prior slices already bled on (entry guards, newline ordering, denylist env, tests that can't fail, the degradation law, verify-on-the-floor). Reusing those fixes is cheaper than rediscovering them.


This is the slice that makes `npx coreartifact init` a real sentence. It is
`med` risk: it touches `package.json`, which is the file that decides what a
user actually receives, and a packaging mistake ships a CLI that cannot find
its own hook artifact.

## What it delivers

- **Two bins**: `coreartifact` and `cart`. `cart` is the short alias and is
  behaviorally identical — same entry, same argv handling. `cart log` output is
  byte-identical to `coreartifact log`.
- A **`files` list** (or equivalent) that ships exactly what the CLI needs at
  runtime: the built CLI, and — critically — **the built hook artifact**. If the
  artifact is not in the tarball, `init` from an installed package has nothing
  to copy and the whole product is dead on arrival. This is the failure mode the
  second criterion exists to catch.
- A **build that runs before pack**, so the tarball contains built output rather
  than sources.

## The npx path, checked honestly

The criterion is checked via **pack + install into a tmpdir**, not against the
live npm registry. That is deliberate and sufficient: it proves the tarball's
contents, its bin wiring and its runtime self-sufficiency, which is everything
`npx` depends on.

**`private: true` stays in `package.json`.** It is the guard against publishing
before the `coreartifact` name is reserved (an operator act, deferred). Do not
remove it — `npm pack` / `pnpm pack` work fine with it set. Removing that guard
to "make packaging work" would be the single worst change in this campaign.

## Any change to package.json changes the lockfile

They move together. An issue that touches `package.json` touches
`pnpm-lock.yaml` too. pnpm only — never npm, never yarn.

## Invariants (contract prose, not acceptance)

- `private: true` remains set.
- Nothing leaves the machine: no postinstall script that phones home, no
  telemetry, no analytics. A postinstall script at all is a smell here — init is
  explicit and user-invoked by design.
- The hook artifact stays zero-dependency: packaging must not "helpfully" bundle
  it against `node_modules`.

## Test-harness contract

Copy the `tests/acceptance/` harness from the harness issue **verbatim** — the
tmpdir-repo factory and its isolated HOME and registry.

The acceptance test at `tests/acceptance/packaging.test.ts` does the real thing:

1. `pnpm pack` the repo into a tmpdir.
2. Install the tarball into a **separate** tmpdir package (a bare
   `package.json`, then install the tarball path) so the bins land in that
   tmpdir's `node_modules/.bin`.
3. Create a tmpdir git repo with the harness factory, run the installed
   `coreartifact` bin's `init` in it, assert exit 0 and that the hook artifact
   landed in the repo.
4. Run both `coreartifact log` and `cart log` from the installed bins against
   the same repo and assert their stdout is identical.

This test is slower than the rest (a real pack and a real install). That is the
price of proving the thing users actually run. Keep it to one test file and do
not sprinkle pack-based setup into other slices.

The other acceptance tests invoke the built CLI entry by path via `node` (the
harness's CLI runner) rather than through installed bins — that separation is
intentional, and this issue must not change it.
