# Ingest: the spool becomes the ledger, idempotently and honestly

Ingest is `high` risk because it is where data integrity is won or lost. Its
whole job is to project ground truth (the spool) into a disposable cache (the
ledger) without ever fabricating, losing, mutating or double-counting anything.

Ingest is **lazy**: it runs at read time, triggered by `log` (and later
`show`). Capture is hot-path and dumb; ingest is lazy and smart. This issue
owns the ingest module **and** the `log` command as its trigger — `log` here
ingests and prints one line per session with the required fields. The
**rendering** of that output (formatting, absent markers, cross-repo union,
the worktree gap warning) is the log slice's, which touches this command file
afterwards. Keep the command thin: ingest, then hand the result to a renderer
seam.

## The ingest algorithm (contract)

Run inside a **single transaction**, so event inserts and the cursor advance
commit atomically — a crash rolls back both, never leaving partial state.

1. If the ledger is absent, create schema v1 and the `meta` row
   (`schema_version = 1`, `ingested_bytes = 0`, `lines_seen = 0`).
2. Seek the spool to `ingested_bytes`. Read line by line.
3. For each **complete** line (terminated by a newline):
   `line_no = lines_seen + 1`, then `lines_seen += 1`.
   - **Parse fails** (not envelope v1, or malformed JSON): increment a skipped
     counter, **name the line in the ingest report**, insert no event row — but
     still advance `lines_seen` and `ingested_bytes` past it. The corrupt line
     is permanently skipped and permanently occupies its ordinal. Continue.
   - **Parse succeeds**: promote fields, `INSERT INTO events ... ON
     CONFLICT(line_no) DO NOTHING`, and upsert the session aggregate.
4. Stop at the last complete line. **Leave a trailing partial line unconsumed**
   — a hook may be mid-append. Set `ingested_bytes` to the offset after the
   last consumed newline; persist `lines_seen`.
5. Recompute derived `sessions` columns (facets and `status`) for **all**
   sessions — see Status.
6. Persist `meta` with `last_ingest_at`.

### Idempotency: high-water mark, floored by the unique key

**Chosen: an ingest high-water mark in `meta`, backed by the `line_no` primary
key as a correctness floor.** Both earn their place:

- `ingested_bytes` gives **O(new bytes)** ingest. The spool never rotates this
  campaign, so a full rescan on every `log` would be an unbounded, growing cost
  on the hot read path.
- The `line_no` primary key guarantees **correctness** even when the cursor is
  stale, reset, or the spool is re-read from zero (the rebuild case). A bare
  cursor is fragile: a partial write or a crash between insert and cursor
  persist would double-insert.

*Rejected: full rescan plus content dedupe on every ingest* — correct but
O(spool) forever, and content dedupe has no reliable key (see below).

`lines_seen` persists the physical-line counter so `line_no` numbering
continues correctly across incremental runs **and includes skipped-corrupt
lines**. Without it, `max(line_no) + 1` would silently re-number after a
corrupt line and break rebuild-equivalence.

### Event identity is the spool line ordinal

An event's identity is `line_no`: the 1-based ordinal of its source line in the
spool. Hook payloads carry **no globally unique event id**; timestamps collide,
and the same hook event name legitimately recurs in a session. The ordinal is a
pure function of spool content and position, so a from-scratch rebuild
re-derives it exactly — which is what makes "delete the ledger and re-ingest
rebuilds equivalent rows" true rather than aspirational.

`seq` is a per-session **presentation** ordinal, assigned deterministically in
`line_no` order (a session's Kth event by `line_no` gets `seq = K`). It exists
to give `show` a clean timeline order. It is not the identity.

## Attribution

`sessions.repo_root` and `sessions.worktree_path` come from the core
attribution function: a worktree cwd resolves to the **main** repo root with
the worktree checkout path recorded alongside; a non-git cwd falls back to the
init root. The hook already appended the worktree session's line to the *main*
checkout's spool — that is capture. This issue proves the ledger row reflects
it.

## Status

Recomputed on **every** ingest from the pure derivation in the core contracts,
never a one-way door:

- **closed-clean** — a SessionEnd was captured (`ended_at` is not null).
- **closed-inferred** — no SessionEnd, `last_event_at` older than the staleness
  threshold (the named 12h constant).
- **open** — no SessionEnd, `last_event_at` recent.

Status depends on **wall-clock now**, not only on new spool lines: an `open`
session becomes `closed-inferred` on an ingest that reads *zero* new lines,
purely because time passed. So step 5 recomputes status for **all** sessions,
not only those touched by new events. This is an `UPDATE` — it changes no row
counts, so idempotency holds.

A late-ingested SessionEnd flips `closed-inferred` back to `closed-clean`
automatically, because status is recomputed rather than stored once. SessionEnd
`reason` does not discriminate (observed: `"other"` for both clean completion
and SIGTERM) — presence of the event is the only signal.

## Facets and the degradation law

**NULL means ABSENT.** A facet whose source was unavailable is NULL — never an
empty string, never zero, never a plausible default. The ledger does not
fabricate evidence.

- **shas** — `git.head` from the SessionStart / SessionEnd boundary envelopes.
  A crash with no SessionEnd leaves `sha_after` NULL. A repo with no HEAD reads
  ABSENT, never `''`.
- **footprint** — the one **materialized** facet: distinct file paths touched
  via *file-editing* tool events (Edit, Write and kin). Bash side-effects on
  files are **not** footprint in v1. It is materialized because it is a
  set-valued cross-event aggregate needed by both the log count and the show
  listing; recompute it per touched session each ingest, fully rebuildable from
  `events`.
- **kind** — populated **iff** the recording pass found a fixture-verified
  discriminating signal in real payloads. The smoke test found none, so the
  expected state is NULL for every session this campaign. **Never inferred
  heuristically** — no "it had no UserPromptSubmit so it must be headless." If
  the recording pass named a signal, read exactly that field; if it named none,
  the column stays NULL and the ingest code contains no kind inference at all.
- **command, outcome, duration** — deliberately **not** stored columns. They are
  per-event facets read from the verbatim payload at render time. Promoting them
  would duplicate the verbatim source for no query win. The line is: promote
  nesting keys (`prompt_id`, `agent_id`, `agent_type`, `tool_use_id`) and
  set-valued aggregates (footprint); leave per-event facets in the payload.
- **git.dirty** — captured in the spool on boundary lines, but **no column** in
  v1: no requirement renders it. Recoverable from the spool if a later facet
  needs it. Do not speculatively add it.

## The ingest report

Ingest returns a structured report, not printed strings: rows inserted,
**lines skipped with their line numbers** (the corrupt-line criterion demands
they are *named* in output), and the warnings list (the worktree-gap warning is
filled by the log slice). The command prints it. Keep the report a typed value
so the log renderer can present it without re-deriving anything, and so this
issue's tests can assert on the report's content rather than on formatting that
a later slice legitimately changes.

## Invariants (contract prose, not acceptance)

- **The spool is never mutated by ingest** — not truncated, not rewritten, not
  rotated, not compacted. Ingest opens it read-only.
- Payloads are stored verbatim: `events.payload` holds the envelope's `event`
  member as **source text**, never a re-serialization (key order would drift).
  The envelope's `git` sibling is *not* part of `payload` — it is consumed into
  `sha_before` / `sha_after`.
- Transcripts are referenced by the path already inside the payload; no
  transcript is read or copied this campaign.
- Nothing leaves the machine.

## Test-harness contract

Copy the `tests/acceptance/` harness from the harness issue **verbatim** —
tmpdir-repo factory (with its isolated HOME and registry override), CLI runner,
fixture replayer, worktree helper, and its ledger row readers. Do not
re-implement SQLite access in the test.

Acceptance tests at `tests/acceptance/ingest.test.ts` drive the real seam:
`init` a tmpdir repo, replay a fixture stream through the **installed hook
command** to produce a real spool, then run `log` and assert on **ledger rows**
read back from the real SQLite file.

Specifics the test-author must not have to go discover:

- Fixture streams come from the fixtures issue's loader by scenario name:
  `headless`, `interactive`, `worktree`, `SIGTERM` (SessionEnd present, no
  Stop), `SIGKILL` (stream just stops — the `closed-inferred` case), and the
  hand-authored **corrupt-line** stream.
- The **late SessionEnd** case is built by replaying the SIGKILL stream, running
  `log`, then appending the SessionEnd payload through the hook and running
  `log` again — status must flip `closed-inferred` to `closed-clean`.
- The **staleness** cases need control over the clock relative to the 12h
  constant. Do not sleep and do not mock `Date` at this seam: replay payloads
  whose timestamps place `last_event_at` on either side of the threshold, or
  drive the ingest module's injectable "now" from a unit test. Both are
  acceptable; pick one and say which.
- **Interleaving** uses the replayer's parallel mode: two fixture streams with
  distinct `session_id`s into one spool, asserting the event rows group by
  session id and each session's `seq` is contiguous from 1.
- **Rebuild** = delete the ledger file, run `log` again, and compare the full
  session, event and footprint row sets to the pre-delete snapshot for
  equivalence (same `line_no`, same `seq`, same payloads).
- The **non-git cwd** fallback needs the hook invoked with a cwd outside any
  git repo, with the init root supplied by the installed hook config — the
  session row's `repo_root` must be that init root.

Below-the-seam unit tests under `tests/unit/ingest/` cover pure logic only:
line-ordinal assignment across a corrupt line, footprint set derivation from a
list of tool events, session aggregate folding.
