# coreartifact doctor — the drift reporter

**Read `docs/gotchas.md` first** — entry 1 (no entry-point guards), entry 3
(allowlist env — the controlled-PATH tests live or die on it), entry 5 (the
degradation law).

Doctor is the drift detector's voice: a read-only report of everything
currently degrading and why. It reports, the operator acts — no auto-fix of
any kind. Four sections:

1. **Running Claude Code version.** Execute `claude --version` (resolved
   through PATH) and parse the **first whitespace-separated token** of its
   single output line — the recorded shape is `2.1.211 (Claude Code)`.
   Anything else — multi-line output, a non-semver first token, a missing
   binary, a nonzero exit, a hang (bound the child with a short timeout) —
   renders the running version ABSENT with the standard absent marker.
   Never guess, never scrape deeper.
2. **Tested version range.** A single named exported constant (suggested
   `TESTED_CLAUDE_CODE_RANGE`), value `2.1.208` through `2.1.211` — the
   range the committed fixtures actually span. One constant, one place to
   bump on the next recording pass. (The PRD calls this "the README's
   stamp"; no README exists in the tree yet — the constant is the source of
   truth and any future README renders it, not the reverse.)
3. **Every facet currently ABSENT, with its reason.** Read through the
   absence-record contract's readers across the repo's ledger — session,
   facet, reason, exactly as recorded (the fragile-dependency register's
   enumerated strings). Where a session carries a transcript-derived
   recorded version (populated by enrichment), report it alongside — free
   drift context: "this session was written by 2.1.213, outside the tested
   range" falls out of the data.
4. **Worktree propagation gaps.** Any worktree of the repo missing the
   settings file — reuse the existing worktree-gap module verbatim; do not
   re-derive the scan.

**Exit code:** 0 when nothing degrades — no absences, version obtained,
no worktree gaps. Nonzero when anything does, and the output names each
finding individually (a finding never hides inside a summary count).
Read-only is a law for this command: it does not trigger ledger creation
(a repo with no ledger reports that as a finding, it does not init one),
does not ingest, does not write anywhere. If reporting requires fresher
data, the answer is "run log first" — doctor reads what exists.

## Invariants (contract prose, not acceptance)

- Doctor never reads the global operator state and never pings — it is
  repo-scoped and network-silent like everything else.
- ABSENT rendering reuses the existing absent-marker module — one marker
  everywhere.
- The version subprocess is spawned with the allowlist-scrubbed env
  (import the existing scrub; never `{ ...process.env }`).

## Test-harness contract

Copy `tests/acceptance/harness/` verbatim; acceptance at
`tests/acceptance/ISS-0021/` drives the built CLI as a subprocess. The
harness's env construction is an allowlist — PATH is under test control by
design.

Specifics the test-author must not go discover:

- **Present case:** put a shim executable named `claude` first on the
  subprocess PATH, emitting the recorded output shape (exposed through the
  typed fixture loader's claude-version string — do not hardcode a guessed
  shape). Assert doctor reports that version.
- **Absent case:** a PATH containing no `claude` → running version renders
  the absent marker; this alone is a finding? No — decide and pin: an
  unobtainable running version IS a degradation finding (nonzero), because
  doctor cannot vouch for compatibility it cannot see. Name it in output as
  such.
- **Degradations to report** are manufactured with the machinery earlier
  slices shipped: replay a stream WITHOUT transcript substitution → a cost
  absence (`transcript unavailable`); the clean case replays with
  substitution and asserts exit 0 with no findings (shim present, no
  worktree gaps).
- Worktree gap: create a worktree via the harness helper after init,
  delete its propagated settings file, assert doctor names it.
- Unit tests under `tests/unit/doctor/` cover version-line parsing as pure
  logic: the recorded shape, a multi-line output, an empty string, a
  non-semver token — each → parsed token or ABSENT.
