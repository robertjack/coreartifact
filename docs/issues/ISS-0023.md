# Consent + the weekly ping — opt-in, two fields, inert by default

**Read `docs/gotchas.md` first** — entry 3 (allowlist env), entry 5 (the
degradation law — which here reads: silence folds to no).

The one exception to "nothing leaves the machine", and it is walled on every
side: an anonymous weekly ping carrying exactly two fields — version and
install id — sent only after explicit opt-in, defaulting to no, riding only
the CLI entry. Storage is the operator-state fold contract (its module, its
ops, its fold rules — this issue writes and reads through it, never
reimplements it).

## Consent at init (R10)

First `init` on the machine (folded state has no install id):

- Generate the install id (a random UUID — it is the only value that ever
  leaves the machine and must carry no derivable information), append the
  `install` op, then:
  - stdin is a TTY → ask the one question, naming exactly what the ping
    sends (anonymous weekly ping, version + install id). **Default no**: an
    empty answer or anything but an explicit yes records no.
  - no TTY → record no without prompting and without hanging. The fleet
    lane never blocks.
- Append the `consent` op with the answer. Both ops are appended once, ever.
- Subsequent inits (any repo, same machine): the fold has an install id →
  no question, no new install/consent ops. Changing your mind is a
  documented file edit in v1 (append a consent line by hand) — there is no
  consent-management command, and this issue must not add one.

## The ping (R11)

On CLI entry (once per invocation, any command — wire it in the dispatcher
path, NOT per-command, and never in the hook artifact):

1. Fold the operator state. No consent → do nothing. This branch must be
   provably silent: with consent off, zero transport calls happen across
   every command.
2. Consent on and `now - last_ping_at > PING_INTERVAL_MS` (named constant,
   exactly seven days; a never-pinged state counts as older) → append the
   `ping` op FIRST, then fire one POST and forget it.
   - **Attempt-time recording is deliberate**: the op closes the weekly
     gate whether or not delivery succeeds, so an unreachable endpoint
     never spam-retries and a second invocation inside the interval sends
     nothing.
   - The payload is a JSON object with exactly two keys: `version` (the
     package's own version) and `install_id`. Nothing else, ever — no
     timestamps, no repo names, no usage metrics. This is a law-level wall;
     treat any third key as a defect.
   - The endpoint is one pinned exported constant on the coreartifact.com
     host (suggested `PING_ENDPOINT = "https://coreartifact.com/ping"`).
     The receiver does not exist yet by design — the client fails silent;
     a wrong URL is correctable any time before launch.
3. Fire-and-forget: transport errors are swallowed entirely — no command's
   stdout, stderr or exit code ever changes because of the ping, and the
   process must not linger on a slow endpoint (bound the attempt; never
   hold the event loop open past command completion).

## The injectable transport

The sender takes a transport (a function receiving url + payload). The real
transport POSTs; construction is owned by the CLI layer. **The test seam:**
when the `COREARTIFACT_PING_SINK` environment variable is set, the CLI
constructs a recording transport that appends `{url, payload}` as one JSON
line to that file instead of touching the network. This is the only
injection mechanism acceptance needs across the subprocess boundary; unit
tests inject transports directly. No test performs real network I/O — the
sandbox has none and the law forbids it anyway.

## Invariants (contract prose, not acceptance)

- The hook artifact is byte-unchanged; nothing on the capture path knows
  the ping exists.
- The install id, once generated, never changes (first-wins fold — the
  contract module guarantees it; do not append a second install op).
- Per-repo uninstall does not touch any of this state.
- With `COREARTIFACT_PING_SINK` unset and consent off, the transport is
  never even constructed — silence by construction, not by filtering.

## Test-harness contract

Copy `tests/acceptance/harness/` verbatim; acceptance at
`tests/acceptance/ISS-0023/`. The harness's isolated HOME + registry-root
override also isolates the operator-state file (same global root — the
contract issue guaranteed it); every test starts from a machine with no
state.

Specifics the test-author must not go discover:

- Consent-off silence: run init (non-TTY → records no), then log, show,
  check and doctor with the sink variable set; assert the sink file has
  zero lines. Asserting through subprocesses means the sink path must be
  passed in the allowlisted child env (extend the harness runner's env
  parameter, not the allowlist default).
- Consent-on: append a consent-yes line to the state file through the
  contract module (the documented hand-edit path), then one command →
  exactly one sink line, payload keys exactly {version, install_id}, url
  equal to the pinned constant; immediate second command → still one line.
- Interval boundary and attempt-time gating are unit territory: the sender
  takes an injectable now and transport in `tests/unit/ping/` — cover
  never-pinged, inside-interval, outside-interval, and throwing transport
  (op still appended, error swallowed).
- The TTY ask is unit-covered through an injected tty/answer seam in
  `tests/unit/install/consent.test.ts` (default-no on empty answer,
  explicit yes, non-TTY auto-no); the acceptance seam covers the non-TTY
  path and asked-once persistence (second init in a second tmpdir repo,
  same HOME: no prompt output, same folded install id).
