# The R9 browser flow + browser-seam harness

The single browser-seam criterion (Testing decisions: "exactly one browser
flow (R9)") and the new Playwright harness it runs on, owned by **one** issue —
the ISS-0003 precedent (seam infrastructure and its first-and-only consumer
land together). This is deliberately merged rather than split into a
"harness" contract plus an "R9" feature: a browser harness's only meaningful
test surface is a browser flow, and the only browser flow this campaign allows
is R9, so splitting would pin one behavior across two issues. It is an L for
that reason and because it is the terminal integration test, depending on both
endpoints, both views, and the `open` command.

**Lane:** dispatch (the R10a chromium-in-sandbox probe passed — CLAUDE.md
records R9 provisionally dispatch-lane, fallback armed). If the probe's
confirmation had failed, this issue would move to the operator lane at the ship
gate and dispatch-side coverage would stay HTTP-seam; that fallback is
pre-named, not discovered.

## Contract

`tests/acceptance/ISS-0032/browser-harness.ts` is the reusable browser-seam
infrastructure: launch headless chromium via Playwright (a devDependency;
chromium preinstalled per the R10a probe — the browser downloads nothing, no
network beyond the loopback sockets the test itself opens), navigate to a URL,
query the DOM, and capture a screenshot to a known artifact path. It is the
browser analogue of `tests/acceptance/harness/`. It lives inside this issue's
own acceptance directory (`tests/acceptance/ISS-0032/`) rather than a shared
`tests/acceptance/browser/` dir — this campaign's only browser flow is R9, so a
single owner is correct; a future browser-seam PRD promotes it to shared prior
art on MAIN at that time, exactly as the CLI harness was.

`tests/acceptance/ISS-0032/browser-flow.test.ts` runs the one flow, driving the
**real served UI** end-to-end. It runs under the loop's own runner (`pnpm test`
= vitest): the test spawns the built CLI (`open --port 0 --no-browser`) as a
subprocess, parses the printed URL, drives chromium against it, and asserts DOM
state — so it is loop-runnable, not operator-run.

The flow (R9): seed the R3+R7 repo (reuse ISS-0028's seeding exactly — real
fixture replay through the hook, real `check` binding for A/B, a hand-authored
kind-ABSENT session and a hand-authored out-of-range-`cc_version` drift
session; all hand-authored `ts`/`started_at` UTC-`Z`). Then:

1. Load the printed URL → the overview shows the headline **"1 of 3"** with the
   failing and unknown-kind sessions surfaced and the **drift banner visible**.
2. Navigate to session A's view → see the **derived-marked cost**, a **check
   badge**, an **ABSENT facet** rendered with the explicit absent marker and its
   recorded reason, and **timeline rows**.
3. Capture a **screenshot** as the evidence artifact.

## Test-harness contract

- Copy `tests/acceptance/harness/` verbatim for the CLI-subprocess + fixture-
  replay + tmpRepo primitives; add the browser primitives in
  `tests/acceptance/ISS-0032/browser-harness.ts` (new seam infra, this issue).
- Seeding is the ISS-0028 real-path seeding — do not invent a shortcut; the
  headline "1 of 3" is only correct if A/B/C/E are seeded to the same kinds and
  bound checks the overview endpoint's own test uses (headless A passing,
  headless B failing, headless C unchecked, kind-ABSENT E, plus a drift
  session). Assert DOM text/nodes, not a pixel diff — the screenshot is the
  evidence artifact, not the assertion (taste is judged against the frozen
  prototype by review, PRD Open risk 3).
- **playwright** is added under `devDependencies` (package.json + lockfile
  change together — both in the footprint). It is pinned (PRD Open risk 6); the
  zero-runtime-deps wall means users never feel it.

## Invariants (reviewer prose, not acceptance)

Loopback only; the browser downloads nothing. The dashboard performs no action
— the flow only reads. No schema change. This issue installs nothing into user
repos and adds nothing to `dependencies`.
