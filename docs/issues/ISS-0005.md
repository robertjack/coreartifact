# init: install per-repo capture, idempotently, and propagate to worktrees

`coreartifact init` is the command that **mutates a user's repo**. That is why
it is `high` risk: everything it writes must be inventoried, minimal, merge-safe
and re-runnable. A user must be able to look at `git status` after `init` and
see nothing they did not consent to.

## What init installs (and nothing else)

Exactly five things, all inventoried on stdout:

1. **Hook config** — merged into the per-repo `.claude/settings.local.json`
   (the install locus: explicit opt-in per repo, teammate-invisible, cleanly
   removable). It subscribes exactly nine events: SessionStart,
   UserPromptSubmit, PreToolUse, PostToolUse, PostToolUseFailure,
   SubagentStart, SubagentStop, Stop, SessionEnd. It does **NOT**
   subscribe WorktreeCreate or WorktreeRemove (2026-07-14 amendment):
   those are delegation hooks — a configured WorktreeCreate hook must
   create the worktree and return its path, so a passive capture
   subscription breaks every worktree-isolated agent spawn in the repo
   (observed; see docs/recording-pass.md). The R1 test asserts both the
   presence of the nine and the absence of the two.

   **PreToolUse is deliberate**, not redundant with PostToolUse: it leaves the
   in-flight command visible when a session dies mid-command. Do not "optimize"
   it away.

   Every subscription points at the hook artifact by **absolute path** and
   supplies the init root, using the invocation contract the hook artifact
   issue defines. Do not invent a second one.

2. **Hook artifact** — copied into the repo's `.coreartifact/` directory. It
   must be reachable even if the package was run via `npx` and its cache is
   later evicted, which is why it is copied rather than referenced in place.

3. **Spool + ledger location** — the `.coreartifact/` directory itself. init
   creates the directory; it does **not** need to create an empty ledger (the
   first ingest does that) but it must report where both live.

4. **Gitignore line** — `.coreartifact/` is gitignored. Append one line;
   never rewrite the user's `.gitignore`.

5. **Registry entry** — the repo root is added to the global registry
   (`~/.coreartifact/registry.jsonl`, respecting the path override the core paths
   module exposes) with an added-at timestamp.

The **repo tree diff shows ONLY those additions.** No config file the user did
not have, no reformatted JSON elsewhere, no stray directories. This is directly
asserted.

## Idempotence and merge

Re-running `init` is a supported, ordinary act — a user will do it after a
version bump. It must:

- exit 0;
- add **no duplicate hook entries** (a subscription already pointing at the
  coreartifact artifact is replaced or left alone, never appended a second
  time);
- add **no duplicate registry entry** (uniqueness by repo root — the registry
  module already guarantees this);
- add no second gitignore line.

And it must **merge, never clobber**: a pre-existing `.claude/settings.local.json`
with unrelated user keys keeps those keys intact, byte-for-byte where possible.
Read, merge the hooks section, write. A user who loses their permissions block
to our installer never trusts us again.

(Uninstall is explicitly out of scope this campaign — it is PRD-0002. But
install must be *shaped* so uninstall can be clean: every write is either a
whole file we own, or one identifiable entry inside a file we don't.)

## Propagation to existing worktrees

A session running in a git worktree does **not** load the main checkout's
`.claude/settings.local.json`: the file is gitignored, absent from the
worktree's checkout, and Claude Code has no git-common-dir fallback (observed,
Claude Code 2.1.208). Without propagation, worktree sessions are silently
uncaptured — capture must exist before attribution can matter.

The mechanism is **two layers** (the WorktreeCreate-hook layer was removed
by the 2026-07-14 amendment — the escalation the specs pre-authorized fired;
see docs/recording-pass.md and the hook artifact issue), and this issue owns
the first:

- **init copies** the settings file into every worktree checkout that already
  exists at install time (enumerate them from git).
- **Ingest warns** about any worktree still missing the file — owned by the log
  slice. A worktree created after `init` — by hand or by the platform — stays
  uncaptured until that warning names it, or until `init` is re-run. Accepted
  residual, softened by an observed fact: worktree-isolated **subagents** are
  captured through the parent session's hooks regardless; the gap is only new
  top-level sessions whose cwd is a post-init worktree.

## Output

stdout is an inventory, not a log: one line per installed thing, naming the
absolute path or the location. It is read by a human deciding whether to trust
us, and by a test asserting we installed exactly what we claimed.

## Invariants (contract prose, not acceptance)

- Nothing leaves the machine: `init` makes no network call and pings no
  telemetry endpoint.
- No write path to the spool or ledger exists outside the CLI and the hook
  artifact.
- `init --global` and user-level hooks are **not** in this campaign (a v1.1
  re-entry). Per-repo only.
- No doctor, no checks, no uninstall.

## Test-harness contract

Copy the `tests/acceptance/` harness from the harness issue **verbatim** — the
tmpdir-repo factory (which supplies the isolated HOME and registry override
every `init` test needs), the CLI runner, the fixture replayer and the worktree
helper.

Acceptance tests at `tests/acceptance/init.test.ts`:

- **R1** — fresh tmpdir repo, run `init`, assert exit 0, assert the stdout
  inventory names all five installed things, and assert the repo tree diff
  (`git status --porcelain` plus the untracked/ignored listing) shows only
  those additions. Parse the written `.claude/settings.local.json` and assert
  the **nine**-event subscription list exactly, **and** the absence of
  WorktreeCreate and WorktreeRemove.
- **R2** — run `init` twice; assert no duplicate hook entries and a single
  registry entry. Then seed a `.claude/settings.local.json` containing an
  unrelated user key **before** `init`, and assert that key survives untouched.
- **R3** — use the harness worktree helper to add a worktree **before** `init`,
  then assert the settings file appears in it. Then replay a payload through
  the installed hook command **with the worktree as cwd** (the replayer takes
  the hook command and cwd) and assert the line lands in the **main
  checkout's** spool — proving the propagated settings capture into the right
  ledger.

The registry is a global file — never write the operator's real one. The
harness's isolated registry path is not optional.

Fixture payloads come from the fixtures issue's loader by scenario name; do not
hand-author a WorktreeCreate payload. If the fixture lacks a shape this issue
needs, that is a routed change to the fixtures issue, never a drive-by edit.
