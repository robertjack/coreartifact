# Fix: the log→show handoff — show goes global, short ids resolve

**Before writing anything, read `docs/gotchas.md`** — the bugs prior slices
already bled on (entry guards, newline ordering, denylist env, tests that can't
fail, the degradation law, verify-on-the-floor). Reusing those fixes is cheaper
than rediscovering them.

## Why this issue exists (integration review, 2026-07-15)

The once-per-campaign integration review found a **cross-slice design mismatch
no per-issue review could see**, and it breaks the product's core demo path:

- `log` (ISS-0007) is **global**: it unions the registry across every
  registered repo and prints an **8-char short id** in column 1.
- `show` (ISS-0008) is **local**: it opens only **cwd's** ledger
  (`getPaths(process.cwd())`) and matches only the **full** session UUID
  (`WHERE session_id = ?`).

Executed consequence: a user reads `909204c2` off `log`'s output and runs
`show 909204c2` → "unknown session". Even with the full UUID, `show` fails
unless the user happens to stand inside the owning repo. **The `log → copy id
→ show` path — the whole point of the two commands — does not work.** Each
slice passed its own review because each was internally consistent; the seam
between them was owned by nobody. That is what this issue fixes.

## The contract

`show <session>` becomes **global and prefix-tolerant**, symmetrical with `log`:

1. **Registry union, like `log`.** Resolve the argument across ALL registered
   repos' ledgers — reuse ISS-0007's union + per-repo isolation mechanics
   (unreachable/corrupt repos warn-and-skip, never sink the lookup; no
   resurrection of deleted repos; runs from any cwd, git or not). Do NOT
   reinvent the union — factor or reuse what `log` does. Extract the shared
   resolution into `src/resolve-session.ts` so both commands lean on one
   module rather than `show` growing a private copy.
2. **The argument is an exact full `session_id` OR a unique prefix.** The
   prefix `log` prints (8 chars) must resolve; longer prefixes and the full
   UUID must too. A prefix shorter than what `log` prints is a usage error —
   never a match-everything wildcard (an empty-string prefix matching every
   session would be the phantom-match bug class again).
3. **Ambiguity fails honestly (degradation law applied to lookup).** If the
   prefix matches more than one session row across the union, exit nonzero and
   list every candidate (full id + repo root). Never silently pick one — a
   timeline for the wrong session is a fabricated receipt. Note the
   REAL ambiguous case that exists today: replaying the same fixture stream
   into two repos yields the SAME session_id in both ledgers — two rows, two
   repos, one id. Listing both (with repos) is correct; picking one silently
   is not. (For a full-id match in multiple repos, the same rule applies.)
4. **Ingest before lookup, per repo,** exactly as `log` does — show's lazy
   ingest must now walk the union, not just cwd.

Rendering is untouched: the header, flat timeline, three-state outcome, and
absent markers from ISS-0008 are correct — this issue changes *which sessions
can be found*, not how they print.

## Test-harness contract

Copy the `tests/acceptance/` harness verbatim (isolated registry, tmpdir-repo
factory, CLI runner, fixture replayer, ledger readers). Fixtures via the
loader. Do not modify `tests/fixtures/**` or unrelated `src/core/*`.

**The acceptance test must exercise the REAL handoff** — this is the seam whose
absence let the bug ship: capture in a repo, run `log`, **parse the short id
out of log's actual stdout** (never read it from the ledger — that was my
demo's blind spot), then run `show <short-id>` from a **non-git** cwd and
assert the timeline prints. The ambiguous case is manufacturable with real
fixtures: replay the SAME stream into two registered repos → one short id, two
rows → assert nonzero exit and both candidates listed with their repos.

Unit tests for the pure resolution logic (exact match, unique prefix,
ambiguous, too-short, unknown) in `tests/unit/resolve-session.test.ts`.

Standing rule: for every test, ask "could this test fail if the bug were
present?" — mutation-check the handoff test by reverting show to cwd-only and
confirming RED.

## Invariants (contract prose, not acceptance)

- `show` remains read-only (writes nothing but ingest's ledger maintenance).
- No transcript reading, no network, nothing leaves the machine.
- The flat timeline stays flat; rendering modules are consumed, not redefined.
- `private: true` remains set; `files: ["dist"]` remains the pack allowlist.
