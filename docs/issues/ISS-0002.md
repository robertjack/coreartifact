# Recording pass: committed, version-stamped hook payload fixtures

**Operator-assisted.** Recording real Claude Code payloads needs the
operator's machine and key; an agent cannot produce them. The operator runs
the recording pass and commits the streams; the implementer builds the
manifest, the loader and the validation test around them. If the streams are
not yet present, this issue is blocked on the operator — say so and stop
rather than fabricating payloads.

This is the empirical input to every acceptance test in the campaign. It is
`high` risk because a wrong fixture silently teaches the whole ledger a wrong
payload shape, and because fixtures are **shared test inputs**: regenerating
one is its own routed change, never a drive-by inside another issue.

## What a fixture is

A **fixture stream** is a file of hook payloads, one JSON object per line, in
the order Claude Code delivered them — the payload *as the hook receives it on
stdin*, not the spool envelope. Downstream tests replay a stream by piping
each line into the installed hook command exactly as Claude Code would.

Five scenarios must be covered:

- **interactive** — a real interactive session, end to end. (The smoke test
  covered only headless; this is the inverse of the usual gap and the operator
  must eyeball one.)
- **headless** — `claude -p`, full lifecycle: SessionStart, UserPromptSubmit,
  PreToolUse, PostToolUse, Stop, SessionEnd.
- **worktree** — a session whose cwd is a git worktree checkout, plus a
  WorktreeCreate event. **The WorktreeCreate payload shape is the campaign's
  first open risk**: does it carry the new worktree's path? The recording pass
  answers it. If the payload does not carry the path, do not invent one —
  record what was observed, and escalate: propagation degrades to init-time
  copy plus an ingest warning, and the second clause of the propagation
  criterion re-opens by escalation, never silently.
- **SIGTERM** — observed: SessionEnd fires, no Stop.
- **SIGKILL** — observed: the stream simply stops. No Stop, no SessionEnd.
  This is the crash path that `closed-inferred` exists for.

Each stream must include the payload variants the facet criteria depend on:
a Bash `PostToolUse` carrying `tool_input.command`, `tool_response`,
`tool_use_id` and `duration_ms`; a `PostToolUseFailure` carrying its `error`
string (which embeds `Exit code N`) and **no** `tool_response`; an
auto-backgrounded `PostToolUse` whose `tool_response` carries a
`backgroundTaskId` with no exit outcome; file-editing tool events (Edit,
Write) that produce a footprint; and Subagent events carrying `agent_id` and
`agent_type` (there is no `subagent_id` — the spawning tool is named `Agent`).

## Version stamping

Every stream is stamped with the exact Claude Code version it was recorded on.
Hook drift across upgrades is a standing, accepted risk: the fixtures pin the
recorded version so payload drift surfaces as *ingest degradation*, never
capture loss (the envelope stores the payload verbatim regardless). The stamp
lives in the manifest, not inside the stream (the stream must remain a
byte-faithful replay input — nothing coreartifact-shaped may leak into it).

## The manifest

A single committed manifest describes the fixture set: for each stream, the
scenario name, the file, the Claude Code version, and the hook event names in
order. It is the one place a later issue looks to find "the headless stream"
or "the worktree stream" by name, so streams can be re-recorded without every
downstream test learning new filenames.

Ship a small typed loader alongside it (load a stream by scenario name; return
its lines). Later acceptance tests import the loader, never a raw path.

## The kind question (open risk 2)

Session **kind** (headless vs interactive) is populated *iff* the recording
pass finds a discriminating signal in the real payloads of the interactive and
headless streams. The pass must answer this explicitly, and the answer must be
written down in `docs/recording-pass.md`:

- If a discriminating field exists, name it, and later ingest reads it.
- If it does not (the expected outcome — the smoke test found none), kind
  records **ABSENT** permanently for this campaign. Never inferred
  heuristically, never defaulted to a plausible value.

Do not let this become a silent default. The recording-pass note is the
evidence that ABSENT is honest rather than lazy.

## Latency (open risk 5)

The pass also measures the hook artifact's per-event latency once that
artifact exists, and records the number in `docs/recording-pass.md`. This
issue's obligation is to leave the section present with the method written
down; user-noticeable overhead re-opens the artifact's form by escalation.

## Corrupt-line fixture

Ingest must skip a corrupt line, count it, name it in output, and keep
ingesting. Give it a real file to do that against: a stream whose middle line
is not valid JSON, with valid payloads on either side. This one is
hand-authored, not recorded — it is the only fixture that may be.

## Invariants (contract prose, not acceptance)

- Fixture streams are byte-faithful replay inputs. They are never edited to
  make a test pass; a failing test against a true fixture is a real defect.
- Payloads are recorded from a real session. No hand-written "plausible"
  payloads, with the single exception of the corrupt-line fixture above.

## Test-harness contract

Plain vitest unit test at `tests/unit/fixtures.test.ts` — this issue predates
the acceptance harness and must not depend on it. The test validates the
manifest against the files on disk (every named stream exists; every line of
every stream is a JSON object carrying `session_id`, `hook_event_name`, `cwd`
and `transcript_path`; the five scenarios are all present; every stream is
version-stamped). It reads files directly, no mocks.
