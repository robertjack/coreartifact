# Hermetic replay by construction: fold the cwd/transcript_path pin into the shared replay harness

**Read first: `docs/gotchas.md` §8 and
`docs/prd/PRD-0003-dashboard/retro.md` "The one process change" — this
issue IS that process change, ruled at the PRD-0003 retro (2026-07-20).**

## Context

Recorded fixture streams embed the recording machine's absolute `cwd`
and `transcript_path`. On the machine that recorded them those leftovers
still exist, so a verbatim replay silently attributes sessions into
stale live repos or enriches a "transcript-absent" session from a real
leftover transcript. This class cost PRD-0003 one escalation (ISS-0028:
spend 0.0558… where the oracle said 0.555957), one re-recording detour
(ISS-0025 fixture authoring), and one latent-exposure finding (ISS-0029
re-confirmation) — three hits on the same nail in one campaign. Today
every acceptance seeding path goes through `replayLines`, but each test
file re-derives (or forgets) the pin by hand: four private
`transformLines` copies, ISS-0028's `pinLineToRepo`, ISS-0029's
`seedLines`. The ruling: move the pin into the harness so the
machine-leftover class is **unexpressible rather than remembered**.

## Contract

- The shared replay primitives (`replayLines`, `replayFixtures`,
  `replayFixturesParallel`) gain a **required** pin target — the test's
  tmp-repo root. An unpinned replay must not be expressible in the type
  system. Every parseable payload line is delivered with `cwd` set to
  the pin target.
- `transcript_path` is pinned to a guaranteed-nonexistent sentinel
  INSIDE the caller's tmpdir unless the caller substitutes one (present
  case: `buildSubstitutedTranscript`'s tmpdir copy; deliberate-missing
  case: an explicit override, ISS-0024 precedent). Parallel replay
  requests each carry their own pin — multi-repo tests (ISS-0007,
  ISS-0012) pin different repos in one file, per call.
- A line that does not parse as JSON (corrupt-line capture-robustness
  corpus) passes through verbatim: never dropped, never edited, never a
  crash. All payload bytes other than the two pinned keys are delivered
  unchanged.
- The doc-comment contract at the top of `fixtureReplayer.ts` currently
  promises "payload bytes unchanged" — update it to state the new
  contract (harness pins `cwd` + `transcript_path`, everything else
  unchanged). `harness.test.ts`'s byte-equality self-tests (including
  the parallel-replayer none-lost/none-duplicated proof) must be
  updated to assert equality post-pin — those failures are contract
  migration, not regressions.
- Delete the per-file helpers and absorb their pin duty:
  `transformLines` cwd-pins in ISS-0019/0021/0024/0025, ISS-0008's
  rebase helper's cwd duty, ISS-0028's `pinLineToRepo`, ISS-0029's
  `seedLines` pin duty. Non-pin duties those helpers also carry (e.g.
  `seedLines`' sessionId override, which keeps concurrent sessions
  distinct) may stay local or become a harness option — implementer's
  choice, but do NOT grow a general payload-transformation API; the
  harness pins, it does not transform.
- Migration of prior issues' tests is mechanical: call sites move to the
  new signature, assertions and oracles do not change. This issue's
  operator sanction to edit locked tests extends exactly that far.

## Boundaries

- Tests-only footprint: no `src/**` changes. The capture law (the hook
  appends verbatim and knows nothing) is untouched — the pin lives in
  the test harness, upstream of the hook's stdin.
- The ISS-0027 default-port test's environmental collision with a
  running dashboard (retro addendum) is a SEPARATE daily-lane item — do
  not fix or touch it here.
- Committed fixture files are only ever read, never rewritten in place
  (standing transcriptReplay rule).
