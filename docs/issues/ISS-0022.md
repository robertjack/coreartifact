# coreartifact uninstall — the way out, byte-identical

**Read `docs/gotchas.md` before writing anything** — entry 4 (assert the
property end-to-end, not a proxy) is the whole acceptance strategy here, and
entry 5 governs every "was this ours?" decision.

Init exists; uninstall does not — the product currently has no way out,
which is `high` risk made worse by what this command does: it mutates user
repos destructively. The bar is exact inversion: after
`init → captured session → uninstall --yes`, the repo tree is
**byte-identical** to its pre-init snapshot. Not "cleaned up" —
byte-identical.

## What init added (each must be exactly reverted)

Invert the install module's own inventory — read the init/install sources in
this issue's footprint and mirror them item by item, so a future init change
breaks THIS list loudly rather than silently:

- **Hook config entries** in the repo's Claude settings file. Two shapes:
  init created the file → remove the file entirely; the file pre-existed →
  remove exactly the entries init merged and preserve every unrelated user
  key byte-for-byte (the same merge discipline init has, inverted).
- **The hook artifact** copy.
- **The spool and ledger** — the whole per-repo `.coreartifact/` directory.
  This deletes captured evidence; that is the command's stated purpose and
  why confirmation is required.
- **The gitignore line** init added. If init created the ignore file, remove
  the file; if it appended to an existing one, remove exactly the line and
  restore the file's prior bytes (including its final-newline state).
- **Propagated worktree copies** of the settings file, with the same
  created-vs-merged inversion per worktree.
- **The registry entry** — by APPENDING one `remove` op for the repo root.
  The registry log is never rewritten, truncated or compacted; append-only
  stays append-only. The fold already honors remove tombstones (shipped
  with the registry module — extend the module with a `removeLedger` append
  helper mirroring `addLedger`: one atomic O_APPEND, no read first).

Idempotence at the edges: uninstall in a repo that was never initialized (or
already uninstalled) exits nonzero saying so and deletes nothing — never a
guess about what to remove.

## Consent

- `--yes` → proceed without prompting.
- No `--yes`, stdin is a TTY → print the full inventory of what will be
  deleted (each path and each settings/gitignore entry), require an explicit
  confirmation, abort cleanly on anything else.
- No `--yes`, no TTY → exit nonzero naming the `--yes` requirement, delete
  nothing, never hang. The fleet lane never blocks on a prompt.

## Invariants (contract prose, not acceptance)

- Per-repo uninstall never touches the global operator state (install id
  and consent are machine-scoped and survive) and never touches any other
  repo's artifacts or registry lines.
- Uninstall never runs git mutations — it removes only files coreartifact
  wrote; the user's working tree, index and history are not its business.
- The pre-existing-settings shape preserves key ORDER and formatting where
  init's merge preserved them — byte-identical is the criterion, so the
  restore path must not re-serialize what it did not change.

## Test-harness contract

Copy `tests/acceptance/harness/` verbatim; acceptance at
`tests/acceptance/ISS-0022/` drives the built CLI as a subprocess in tmpdir
repos with fixture replay through the installed hook command.

Specifics the test-author must not go discover:

- **The snapshot is a full recursive tree capture** (relative path → bytes,
  including dotfiles) taken BEFORE init; after
  init → replay → `log` → `uninstall --yes`, capture again and assert map
  equality. Exclude `.git/` internals from the comparison (git touches its
  own files); everything else compares byte-for-byte. Run it in both
  settings shapes: fresh repo, and a repo pre-seeded with a settings file
  and gitignore carrying unrelated user content.
- The worktree shape: init a repo that already has a worktree (harness
  worktree helper), assert the propagated settings copy is gone after and
  the worktree tree matches its own pre-init snapshot.
- Registry assertions read the raw registry file: prior bytes are a strict
  prefix of the after bytes, exactly one appended line, `op` remove, the
  root named; then assert `log` (run from a non-git cwd) no longer lists
  the repo while a second registered repo still lists.
- The global-state assertion: with the registry-root override in place,
  create a state file with known bytes under the global root before
  uninstall and assert byte-equality after.
- The TTY confirmation path cannot be driven at the subprocess seam (no
  PTY): unit-test the prompt gate below the seam with an injected
  is-TTY/answer seam in `tests/unit/install/uninstall.test.ts`; the
  acceptance seam asserts the non-TTY refusal and the `--yes` path.
