# show: the flat timeline, and the three-state outcome

`show <session>` is where the evidence actually reads as evidence. It is the
slice that must make **absence legible**: this is the campaign's product
promise, and the one guarantee the ledger must never violate тАФ it does not
fabricate evidence.

## What show prints

A **header** followed by a **flat chronological timeline**.

Header: the session's `sha_before` and `sha_after`, and its footprint (the
distinct file paths). Absent shas render as the explicit absent marker.

Timeline, ordered by the per-session `seq` (the presentation ordinal ingest
assigned in spool-line order тАФ do not re-sort by timestamp; timestamps collide):

- lifecycle events (SessionStart, Stop, SessionEnd, and the rest);
- prompts (UserPromptSubmit);
- **every command** with its command string, **outcome** and **duration**;
- subagent events, carrying `agent_id` and `agent_type`.

**Flat, not a tree.** The nesting keys (`prompt_id`, `agent_id`, `agent_type`,
`tool_use_id`) are captured and available, but v1 renders a flat stream. The
tree view is v2 and is pure UI when it arrives тАФ no migration. Do not indent by
`agent_id`, do not group. Resist it.

An **unknown session id** exits nonzero with an error **naming the id**.

## Facets are derived at render, from the verbatim payload

Command string, outcome and duration are **not** ledger columns. They are
per-event facets read from `events.payload` (the verbatim hook payload) at
render time. This issue owns that derivation module. What it reads, from the
observed payload shapes:

- **command string** тАФ Bash `PostToolUse` `tool_input.command`.
- **duration** тАФ `duration_ms`, present on both `PostToolUse` and
  `PostToolUseFailure`.
- **shas** тАФ `sessions.sha_before` / `sha_after` (promoted by ingest from the
  boundary envelopes' `git.head`).
- **footprint** тАФ the `footprint` table (distinct paths from file-editing tool
  events; Bash side-effects on files are not footprint in v1).

## The three-state outcome тАФ the heart of this issue

Outcome has **three** states, and they must never collapse into two:

- **success** тАФ `PostToolUse` on a Bash command, no failure marker.
- **failure** тАФ `PostToolUseFailure`, whose `error` string is **preserved
  verbatim** in the output (it embeds `Exit code N` plus the message).
  Observed: `PostToolUseFailure` carries **no** `tool_response` тАФ do not read
  one.
- **ABSENT** тАФ an **auto-backgrounded** command: `PostToolUse` whose
  `tool_response` carries a `backgroundTaskId` with no exit outcome (observed:
  a long Bash was auto-backgrounded, PostToolUse fired in 155ms with empty
  stdout). Its outcome is **ABSENT** and must render as the absent marker,
  distinguishable from both success and failure.

There is **no reaction to backgrounded-command completion** this campaign:
outcome-absent is final. Whether a later event ever carries the eventual outcome
is unverified; it is revisited in PRD-0002. Do not go looking for one, and do
not optimistically render it as success тАФ that would manufacture a false
receipt, which is exactly the failure mode the whole product exists to prevent.

## The absent marker (shared surface, routed)

Import the absent marker from the module the log slice owns; do not define a
second one. This issue **touches** `src/render/log.ts` only if the R12
assertions for `log` (sha-absent, kind-absent) require a rendering fix there.
That is a routed amendment: the log slice's own tests must stay green.

## Invariants (contract prose, not acceptance)

- Payloads are read, never rewritten; `show` writes nothing but the ledger the
  ingest module maintains.
- Transcripts are referenced by path, never read or copied тАФ even though the
  path is in the payload. No transcript reading at all this campaign; cost
  enrichment is PRD-0002.
- The canonical nesting key is `agent_id` (with `agent_type`). There is no
  `subagent_id`. The spawning tool is named `Agent` in Claude Code 2.1.208.
- Nothing leaves the machine.

## Test-harness contract

Copy the `tests/acceptance/` harness from the harness issue **verbatim** тАФ
tmpdir-repo factory with its isolated registry, CLI runner, fixture replayer,
worktree helper, ledger readers.

Acceptance tests at `tests/acceptance/show.test.ts` drive the real seam: `init`
a tmpdir repo, replay a fixture stream through the installed hook command, run
`log` (to ingest), then run `show <session>` and assert on stdout.

Specifics the test-author must not have to go discover:

- The session id comes from the replayed fixture's payloads (every payload
  carries `session_id`) тАФ read it from the fixture, never hardcode it.
- The **failure** case needs the fixture's `PostToolUseFailure` event; the
  **outcome-absent** case needs its auto-backgrounded `PostToolUse` (the one
  whose `tool_response` carries `backgroundTaskId`); the **subagent** case needs
  its SubagentStart/SubagentStop plus a tool event carrying `agent_id` /
  `agent_type`. All three are guaranteed present in the recorded streams by the
  fixtures issue. If one is missing, that is a routed change to the fixtures
  issue тАФ never a hand-authored payload here, and never a fixture edited to make
  a test pass.
- **sha-absent** is produced by the SIGKILL stream (no SessionEnd, so
  `sha_after` is NULL).
- **kind-absent** is asserted against `log` output (kind is expected NULL for
  every session this campaign).
- **Unknown session id** тАФ run `show` with an id no fixture carries; assert
  nonzero exit and that the id appears in the error.

Below-the-seam unit tests under `tests/unit/facets/` cover the outcome
derivation as pure logic: given a payload with `backgroundTaskId` тЖТ ABSENT;
given a `PostToolUseFailure` with an `error` тЖТ failure with the error preserved;
given a plain Bash `PostToolUse` тЖТ success. Three states, three payload
signatures, no overlap.
