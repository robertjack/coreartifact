# open command + dashboard server core (the GET wall)

The HTTP front door of the dashboard: a `node:http` server, the `open` CLI
command that launches it, and the strictly-read-only GET wall. This is one
issue because at the seam the two are inseparable — every dashboard
acceptance test spawns `coreartifact open --port 0 --no-browser` as a
subprocess and asserts raw HTTP, so the server cannot be red-tested without
the command, and the command does nothing without the server. It is an L for
that reason; splitting would leave a contract with no test surface.

## Contract

`src/dashboard/server.ts` exports a function that creates and starts the
`node:http` server and resolves to `{ url, close() }`. It:

- **Binds loopback only.** Listen on a loopback host; never `0.0.0.0`.
- **Port selection.** With an explicit port, use it (port `0` = OS-assigned
  ephemeral, the acceptance lane). Without one, try `DASHBOARD_DEFAULT_PORT`
  (**2278**); on `EADDRINUSE`, fall back to an ephemeral port. The **printed
  URL is authoritative** either way — derive it from the actually-bound
  address, never from the requested port.
- **Lifecycle.** SIGINT and SIGTERM close the server and let the process exit
  cleanly, releasing the port. No lingering handles.

`src/dashboard/routes.ts` is the **route registry** — the shared surface both
endpoint issues extend, mirroring `src/cli/index.ts`'s `COMMANDS` table. It
maps `/api/overview` and `/api/session/<id>` to handler functions, shipping
here with **stub handlers** (a valid empty-shaped 200 JSON with
`Cache-Control: no-store`) so the server is green and the GET wall is testable
before any endpoint lands. ISS-0028 and ISS-0029 each touch this file to wire
their real handler in place of a stub (one entry each, non-conflicting — the
gm-portal green-alone-throws-together surface, resolved by the registry
pattern).

`src/dashboard/constants.ts` owns every published dashboard constant (api.md
"Named constants"), imported by the endpoint issues:
`DASHBOARD_DEFAULT_PORT = 2278`, `LATEST_SESSIONS_LIMIT = 50`,
`OVERVIEW_WINDOW_DAYS = 7`, `READ_BUSY_TIMEOUT_MS = 5000`, and the loopback
host allowlist (`localhost`, `127.0.0.1`, `[::1]`/`::1`; any `:port` suffix
allowed).

`src/dashboard/assets.ts` serves the built SPA out of **`dist/web/`** (the
asset-root contract ISS-0026 established). Resolve the asset root relative to
the compiled server module's own location (`dist/dashboard/server.js` →
`../web`). Content-hashed assets may carry long-lived/immutable caching; the
shell and every `/api/*` response carry `Cache-Control: no-store`.

`src/cli/commands/open.ts` parses `--port <n>` and `--no-browser`, starts the
server, prints the bound URL on stdout, and installs the signal handlers.
Browser auto-open happens **only** when stdout is a TTY and `--no-browser` is
absent (open the printed URL via the platform opener, allowlist-scrubbed env
per `src/core/attribution.ts` — never `{ ...process.env }`, gotcha #3). It
registers in `src/cli/index.ts`'s `COMMANDS` table as a one-line entry (the
established shared-surface amendment; `cart` gets it for free — both bins
point at the same `bin.js`).

## The GET wall (api.md Surface A)

Order the checks so nothing attacker-controlled is ever reflected or served:

1. **Method** not GET/HEAD → `405`, `Allow: GET, HEAD`, body code
   `method_not_allowed`.
2. **Host** header host-part not in the loopback allowlist (or absent/empty)
   → `403`, body code `forbidden_host`, reflecting no host value. This is the
   DNS-rebinding wall — a second layer independent of the loopback bind.
3. **Path traversal** — a resolved path outside the asset root → `404`, body
   code `not_found`, naming the path. Never yields content outside
   `dist/web/`.
4. Route: `/api/*` → the registry; a real asset file → served; any other
   non-`/api` path → the shell (SPA client-routing fallback, `200`).

Error bodies use the single api.md shape everywhere:
`{ "error": { "code": "<machine_code>", "message": "<human>" } }`,
`Content-Type: application/json; charset=utf-8`.

## Test-harness contract

- Copy the acceptance harness at `tests/acceptance/harness/` (import from it;
  never fork it). Spawn the built CLI via `runCli` with `open --port 0
  --no-browser`, parse the bound URL from stdout, and drive raw HTTP with
  `node:http` requests on loopback (no fetch polyfill needed; assert status,
  headers, and body bytes). Real sockets, zero mocks at the seam.
- Register a repo first (the harness's tmpRepo + `init` path) so "with a
  registered repo present" holds; `GET /` must return the real
  `dist/web/index.html` shell (this is why ISS-0026 must land first and why
  globalSetup builds the SPA).
- Prove the port fallback by binding 2278 yourself first (open a listener on
  it) and asserting the printed URL carries a different port — do not assert a
  specific ephemeral value.
- Prove clean shutdown by sending the real signal to the child process and
  asserting the process exits and the port is re-bindable afterward (gotcha
  #4: exercise the real process, assert the property end-to-end).
- The `/api` stub is enough to assert the `no-store` header and the method/
  host/traversal walls at this issue; the real endpoint bodies arrive in
  ISS-0028/0029.

## Invariants (reviewer prose, not acceptance)

Strictly read-only: the server mutates nothing but the sanctioned ingest
projection (and this issue does not even read ledgers yet). Nothing binds
beyond loopback. The hook artifact, spool, ledger, and `log`/`show` output are
untouched. `bin.ts` stays guardless (gotcha #1); `open.ts` never guards an
entry point.
