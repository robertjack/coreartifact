# SPA scaffold + build wiring into dist

Establish the vite+react single-page-app project and wire its build into the
package so every later dashboard issue builds against a settled UI shell,
asset location, and build pipeline. This issue writes no product behavior —
it is the enabling infrastructure the frozen prototype and both view issues
inherit and never re-wire. It owns no PRD requirement criterion; its
acceptance is the build/artifact contract the rest of the campaign depends on.

## What this delivers

- A vite+react project rooted at `web/` (outside `src/`, so the existing
  `tsc` build — `rootDir: src`, `include: ["src/**/*"]` — never touches it).
  The app shell renders the dashboard chrome and a client-side router with
  exactly two routes: the overview (`/`) and the session view
  (`/session/:id`). Each route renders a **stub view component the later UI
  issues own and replace** (see Seams).
- `web/src/api-types.ts`: the TypeScript shapes of the two API responses,
  transcribed from `docs/prd/PRD-0003-dashboard/api.md` (the overview and
  session-view JSON). This is the single place the UI issues import the
  contract from; the server side encodes the same shapes independently (the
  coupling is api.md, not a shared module across the tsc/vite boundary).
- The styling toolchain frozen in PRD Amendment 2, devDependencies only:
  Tailwind 4 via `@tailwindcss/vite`, shadcn/ui primitives generated from
  shadcn's own MIT source via its CLI into `web/src/components/ui/`, lucide
  icons, neutral base. No charting/visualization library, no theming, no
  dark-mode toggle (PRD non-goals). The premium reference template stays
  gitignored and license-walled — visual direction only, zero code reuse.
- `pnpm run build` grows an SPA build step so it produces **both** the CLI
  (`tsc` → `dist/cli/…`) **and** the SPA (`vite build` → `dist/web/…`).
  The vite build output directory is **`dist/web/`** and is the binding
  asset-root contract every server issue serves from. `package.json`'s
  `files: ["dist"]` already ships it; do not remove `private: true`.

## The build-seam amendment (operator pre-dispatch, on MAIN — do not skip)

`tests/acceptance/harness/globalSetup.ts` today execs `tsc` **directly**
(`node node_modules/typescript/bin/tsc`), NOT `pnpm run build`. Every
dashboard acceptance test spawns the built CLI as a subprocess and the server
serves `dist/web/`; if globalSetup only runs `tsc`, `dist/web/` never exists
at test time and every HTTP-seam test fails to find the shell. The shared
acceptance harness must be amended to build the SPA as well (run the full
`pnpm run build`, or add an explicit `vite build` exec after the `tsc` exec),
keeping the "build exactly once in the root process before any worker forks"
property intact (the S1b finding) — do not move the build into a per-worker
path.

Because `tests/acceptance/harness/` is shared prior-art harness infrastructure
(not any one issue's own acceptance directory), that edit is an **operator
pre-dispatch amendment on MAIN**: it is a cross-cutting harness change that
lies outside every issue's footprint, exactly the class the decomposer routes
to hand-work on MAIN. It is landed once, on MAIN, before this campaign's
HTTP-seam issues (ISS-0027/0028/0029/0032) dispatch; every dashboard
acceptance issue then inherits it, and no issue owns or touches
`globalSetup.ts` in its footprint. This issue's own `pnpm run build` gate and
`scaffold-build.test.ts` prove the build produces `dist/web/` independently of
the harness amendment.

## Seams and shared surfaces

- **Router extension points (stub created here, owned by the UI issues).**
  This issue creates `web/src/views/Overview.tsx` and `web/src/views/Session.tsx`
  as minimal stubs — carried in this issue's `touches`, because the durable
  owners are the UI issues — and wires the router (in `web/src/App.tsx`, owned
  here) to them once, so this scaffold builds green on its own. ISS-0030
  **owns** and replaces `web/src/views/Overview.tsx` (plus
  `web/src/views/overview/**`); ISS-0031 **owns** and replaces
  `web/src/views/Session.tsx` (plus `web/src/views/session/**`). Because each
  view file's durable owner is exactly one issue and the router registration is
  never edited by the UI issues, the two view issues never collide on the
  router (the shared-surface flag the PRD raised is resolved by ownership, not
  by both issues editing one file).
- **package.json + lockfile change together.** Adding devDependencies edits
  both `package.json` and `pnpm-lock.yaml`; both are in the footprint.
- **.gitignore** gains the premium reference template path and any vite/node
  build artifacts that must not be committed.

## Test-harness contract

- Acceptance is a single vitest unit (`tests/unit/scaffold-build.test.ts`) that
  runs `pnpm run build` (or asserts against its already-built output) and
  checks the built artifacts exist: `dist/web/index.html` present and
  non-empty, an `assets/`-hashed script referenced by the shell. Do not assert
  exact hashed filenames (they change with content — gotcha #7: assert
  containment/existence, never today's exact snapshot).
- Judge all visual work against `docs/prd/PRD-0003-dashboard/prototypes/v2-tile-led.html`
  (the frozen design contract). Taste deltas after the freeze are operator
  amendments, not review rounds.

## Invariants (reviewer prose, not acceptance)

Zero runtime dependencies: nothing added to `dependencies`; `private: true`
stays in package.json (the publish guard). The ledger, spool, and hook
artifact are untouched. `pnpm run typecheck` remains `tsc --noEmit` over
`src/` only and must not attempt to compile anything under `web/` (vite owns
the SPA's own type-check at build). This issue installs nothing into user
repos and touches no capture/ingest path.
