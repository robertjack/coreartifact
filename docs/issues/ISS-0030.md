# Overview UI — headline, tiles, session list, drift banner, empty/unreadable states

The React overview view, built against the frozen `api.md` overview shape
(imported from `web/src/api-types.ts`, established by ISS-0026) and judged
against the frozen design contract
`docs/prd/PRD-0003-dashboard/prototypes/v2-tile-led.html`. This issue replaces
the scaffold's `web/src/views/Overview.tsx` stub; the router wiring already
points at it (no router edit — the shared surface was resolved by ownership in
ISS-0026).

## Contract

Render the v2-tile-led layout from a fetched `/api/overview` response:

- **KPI hero card** — the verified-delegation headline as "verified of
  delegated_total" (e.g. "1 of 3"), with the three-way split as a proportional
  **segmented bar** over `verified` / `failing` / `unverified`. Counts come
  from the API; the UI computes the ratio and bar geometry (the API returns
  counts only, never a ratio — a zero denominator renders as a real empty
  state, never NaN).
- **Stat tiles beneath** — spend (`spend_present_usd`, labeled **derived**;
  when `cost_absent_count > 0`, surface "unknown for N sessions" rather than
  implying the sum is complete — the anti-silent-zero discipline on the wire,
  made visible), sessions-by-kind, failing-checks. Numbers only — no charting
  library, no theming, no dark-mode toggle (PRD non-goals).
- **Session list** — roomy list-rows over `sessions.latest`, each linking to
  its session view. Links **always carry `?repo=<root>`** (from the entry's
  `repo_root`) so the session endpoint never hits the ambiguous-id case. A row
  whose `classification` is `null` omits the three-way badge (interactive and
  unknown-kind sessions are not classified — never default the badge).
- **Drift banner** — rendered from the `drift` array, naming each session, its
  version, and the range; **absent when the array is empty** (fires on positive
  evidence only).
- **Degraded / empty states** — a `repos` entry with `status: "unreadable"`
  renders as a visible degraded row with its reason (never silently skipped);
  `repos_skipped > 0` surfaces the count; an empty registry renders an explicit
  empty overview (real zeros, not an error screen).

## ABSENT rendering (UI law, PRD Amendment 2)

Reasoned-ABSENT (a `null` with an `absences` row on the wire) renders as V2's
**disclosure chip** — a visible marker whose recorded reason opens inline,
keyboard-operable, no hover dependency. Self-describing nullables (e.g. a bare
null cost with no reason, main-checkout worktree) render **quiet** (a plain dash
/ "main checkout"), per api.md B1.3. Loud vs quiet is the distinction; never
fabricate either way. The overview surfaces cost-ABSENT via the tile's
absent-count, not a per-row chip.

## Test-harness contract

Extract the presentational logic into pure functions under
`web/src/views/overview/` (e.g. a `model.ts` exporting `headline(kpi)`,
`barSegments(kpi)`, `spendTile(tiles)`, `driftBanner(drift)`,
`sessionRow(entry)`) that take the api.md response shapes and return
plain view-model values with **no React or DOM dependency**. Acceptance is a
vitest unit (`tests/unit/overview-view-model.test.ts`, plain `.ts`) over those
functions with fixture props — the sanctioned "unit tests below the seam only
for pure logic" lane. The React components (`.tsx`) consume these functions and
are verified end-to-end by the one browser flow (ISS-0032) and by review
against the frozen prototype; they are not tested through a new component
harness (no jsdom/react-testing-library is introduced — that is scope the PRD
did not authorize).

## Invariants (reviewer prose, not acceptance)

The premium reference template is visual direction only — zero code reuse,
license-walled (a reviewer check on this slice). No UI action of any kind: the
overview is a viewer; the first UI action, when demand appears, is a CLI
invocation (spec ruling), not this campaign. No search, no filters beyond the
`repo` param the API already owns, no pagination beyond the latest-N cap with
its honest total.
