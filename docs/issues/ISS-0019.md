# Cost enrichment — the one transcript-derived facet, fail-soft

**Read `docs/gotchas.md` before writing anything** — entry 5 (the
degradation law) is this issue's spine, and entry 6 (execute, don't assert)
is how the oracles below were validated.

This is the fragile-dependency register's most brittle entry: the transcript
JSONL schema is documented-as-internal and can drift on any Claude Code
release. Enrichment therefore fails soft at every branch — ABSENT with a
named reason, never zero, never estimated, never a crashed ingest — and
derives everything from the transcript **read in place at the session's
stored transcript path, never copied** (law; no transcript column exists).
The trust spine (commands, checks, footprint, shas) never depends on this
facet.

Complexity is `l` and deliberately unsplit: tokens, cost, and the cost
absence reasons are one fail-soft pass whose acceptance pins both the happy
path (oracle reproduction) and every degradation of the same call sequence.
Splitting tokens from cost would let the earlier half's tests pin an
enrichment sequence the later half must change.

## The two layers (both NULL = ABSENT, stored separately)

1. **Token counts — parsed exactly.** Usage rides `type: "assistant"`
   transcript lines at `.message.usage` (`input_tokens`, `output_tokens`,
   `cache_read_input_tokens`, `cache_creation_input_tokens`).
   **Multiple assistant lines repeat one request's usage under a shared
   `requestId`** — a naive per-line summer over-counts about 2.4x (the
   recovered headless pair has 17 assistant lines over 7 distinct
   requestIds, committed as the dedup shape fixture). The pinned rule:
   dedup by `requestId`, take usage once per request, sum across requests.
   Four columns, one per token class, because the price table rates each
   class differently. Unknown transcript line types (system, attachment,
   file-history-snapshot, mode, ai-title, and anything future) are skipped
   silently — tolerate the zoo, parse only what is pinned.
2. **cost_usd — computed, labeled derived.** No dollar figure exists
   anywhere in the transcript (exhaustively verified); dollars are computed
   from a **pinned per-model price table** — a code constant in its own
   module, a new fragile-dependency register entry, never schema. Cost =
   sum over the token classes times the per-class rate for
   `.message.model`, where **cache writes price by TTL**: the transcript's
   per-request `.message.usage.cache_creation` splits
   `ephemeral_5m_input_tokens` / `ephemeral_1h_input_tokens`, and the two
   classes carry different rates — price from that split at parse time
   (the ledger's single cache-creation column is display aggregation,
   never the pricing input). The committed envelope oracles are ground
   truth: the computation must reproduce each `total_cost_usd` to the
   digit.

   **The pinned table (gate ruling 2026-07-16 — do NOT re-derive it; the
   three oracles are 3 equations over 4+ unknowns and underdetermine it):**

   | model id (exact string) | in | out | cache read | write 5m | write 1h |
   |---|---|---|---|---|---|
   | `claude-fable-5` | 10.00 | 50.00 | 1.00 | 12.50 | 20.00 |
   | `claude-haiku-4-5-20251001` | 1.00 | 5.00 | 0.10 | 1.25 | 2.00 |

   USD per MTok. Structural rule, recorded for the next pin: cache read =
   0.1× input, write 5m = 1.25× input, write 1h = 2× input. Provenance:
   fable rates reproduce all three committed oracles exactly (all recorded
   sessions used 1h writes); haiku rates are published pricing at the same
   multiples (the recovered haiku pair has no oracle — shape coverage
   only, and its cost renders like any other computed cost). Other models
   are DELIBERATELY unpinned this campaign — `claude-sonnet-5` sits in an
   intro-pricing window (a pin would fabricate cost when billing
   diverges) and `claude-opus-4-8` has no oracle recording; both degrade
   to tokens-present / cost-ABSENT with `model unpinned: <model>`, which
   is the designed behavior, not a gap. Pin exact model id strings —
   never aliases (repo law).

`model` (the price-table key and display value) and `cc_version` (the
transcript's top-level `version` field — the transcript self-identifies its
writer; doctor renders it later) are stored per session, NULL when
unreadable.

## Degradation states (each representable, each recoverable)

| state | tokens | cost_usd | absence row (facet `cost`) |
|---|---|---|---|
| enriched, model pinned | present | present | none |
| model unpinned | present | NULL | `model unpinned: <model>` |
| transcript missing | NULL | NULL | `transcript unavailable` |
| transcript drifted (parse failed) | NULL | NULL | `transcript shape unrecognized` |

Absence rows go through the absence-record contract module — its enumerated
reason constants, exactly; never ad-hoc strings. Enrichment runs inside
ingest's derived-projection recompute per touched session, is idempotent
(re-run changes no row counts), and **ingest completes normally on every
degradation** — a missing or hostile transcript degrades one session's
facet, nothing more. Because tokens are re-parsed and cost re-computed from
the transcript-at-path on every rebuild, delete-ledger + re-ingest after the
transcript appears (or the price table is fixed) retroactively regains the
facet.

## Render

`log` and `show` gain the cost value rendered **with a derived marker** —
visually distinct from spool-borne facets, because this number is computed
from a pinned table, not observed. Keep it minimal here (the value plus the
marker); the render slice owns the final column/badge layout and the
absent-marker assertions, and touches these files after this issue. Reuse
the existing absent-marker module for the ABSENT case; never print blank or
zero.

## Invariants (contract prose, not acceptance)

- The transcript is read in place; no byte of it is copied, cached, or
  stored in the ledger.
- No estimation, interpolation, or approximate rendering exists anywhere —
  ABSENT or exact, nothing between.
- The hook artifact and the spool are untouched; enrichment is ingest-side
  only and never on the hot path.
- Nothing leaves the machine.

## Render-surface handoff

This issue owns `src/render/log.ts` and `tests/unit/render/**` for the
campaign, and edits `src/render/show.ts` owned by the parser slice — hence
the dependency on it; the render-depth slice edits all three downstream and
depends on this issue. The shared render surface is serialized, never
edited in parallel.

## Test-harness contract

Copy `tests/acceptance/harness/` verbatim; acceptance at
`tests/acceptance/ISS-0019/` drives the built CLI as a subprocess in tmpdir
repos. **Use the fixtures layer's transcript-substituting replay wrapper**
(`tests/fixtures/transcriptReplay.ts`, added by the fixtures issue): it copies the paired transcript into the tmpdir and
rewrites `transcript_path` in delivered payloads — the one sanctioned
substitution; committed fixtures stay byte-verbatim. Load pairs and oracle
values through the typed transcript-pair access, never hardcoded paths; the
expected numbers in acceptance come from the loaded oracle (the literals in
the criteria are the pinned ground truth they must equal).

Specifics the test-author must not go discover:

- Happy path: replay cost-headless with substitution → `log`/`show` carry
  tokens and 0.555957 with the derived marker; ledger columns match the
  oracle via the harness session reader.
- Missing transcript: replay WITHOUT substitution — the recorded
  `transcript_path` does not exist on this machine → ABSENT +
  `transcript unavailable`, ingest exits normally.
- Drifted transcript: substitute, then overwrite the tmpdir COPY (never a
  committed file) with a hand-authored shape outside the pinned parse (e.g.
  assistant lines missing `message.usage`) → ABSENT +
  `transcript shape unrecognized`.
- Recovery: after the missing-transcript run, place the tmpdir transcript
  copy at the recorded path, delete the ledger, rerun → facet present,
  absence row gone.
- Unpinned model: substitute with a tmpdir-copy transcript whose
  `.message.model` is rewritten to an unpinned id → tokens present, cost
  NULL, reason names the model.
- Unit tests (owned files) cover the pure parse: requestId dedup over the
  recovered headless transcript fixture (17 lines, 7 requests — assert the
  naive sum differs), the zoo-tolerant line skipping, price-table lookup
  and the exact oracle arithmetic for all three oracles.
