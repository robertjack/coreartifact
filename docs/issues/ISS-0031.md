# Session view UI — facet header, badges, results, footprint, timeline, absences

The React session view, built against the frozen `api.md` session-view shape
(`web/src/api-types.ts`) and judged against
`docs/prd/PRD-0003-dashboard/prototypes/v2-tile-led.html`. Replaces the
scaffold's `web/src/views/Session.tsx` stub (router already wired by ISS-0026).

## Contract

Render, from a fetched `/api/session/<id>` response (the UI's link carries
`?repo=<root>`):

- **Facet header** — `session_id`, `repo_root`, `worktree_path`, `status`,
  `kind`, shas before/after, `model`, `cc_version`, `started_at`/
  `last_event_at`/`ended_at`, and cost/tokens with the **derived marker**
  (read `.derived` off the value — never hardcode which facets are derived,
  api.md B2). `worktree_path` is shown (a first-class column, PRD Amendment 1).
- **Check badges** — one per bound check: name + passed/failed from
  `exit_code`, truncation shown.
- **Test-results facet** — a present row's counts; an **empty array = ABSENT**
  (no command claimed), distinct from a present zero-count row (a real zero
  run).
- **Footprint** — the touched-paths list; `[]` = a real empty set (not ABSENT).
- **Absences** — the recorded `{ facet, reason }` entries, surfaced.
- **Flat timeline** — rows in spool order, one per entry, surfacing the entry
  kind and its derived facets. A command entry's three-state `outcome` renders
  success / failure(with error) / **absent** faithfully — an `absent` outcome
  (e.g. a backgrounded command with no resolving TaskOutput) shows the explicit
  absent marker, never success. The four nesting keys are surfaced but **not
  interpreted** into a tree (the tree view is a v2 non-goal).

## ABSENT rendering (UI law, PRD Amendment 2)

Reasoned-ABSENT (a `null` with an `absences` row — e.g. `kind`, `cost`) renders
as V2's **disclosure chip**: a visible marker whose reason opens inline,
keyboard-operable, no hover dependency. Self-describing nullables
(`sha_before`/`sha_after`/`ended_at`, a bare `worktree_path` null) render
**quiet** (a plain dash / "main checkout"), per api.md B1.3. Loud vs quiet is
the whole distinction; never fabricate either way, never render ABSENT as `0`.

## Test-harness contract

Extract the presentational logic into pure functions under
`web/src/views/session/` (e.g. `model.ts` exporting `costFacet(facets)`,
`kindMarker(facets, absences)`, `checkBadge(check)`, `testFacet(test_results)`,
`timelineRow(entry)`, `footprintView(footprint)`) taking the api.md session
shape and returning plain view-model values with **no React/DOM dependency**.
Acceptance is a vitest unit (`tests/unit/session-view-model.test.ts`, plain
`.ts`) over those functions with fixture props (the sanctioned pure-logic
lane). The `.tsx` components are verified end-to-end by ISS-0032 and by review
against the frozen prototype; no component-test harness is introduced.

## Invariants (reviewer prose, not acceptance)

No UI action of any kind (viewer only). Premium template is visual direction
only, license-walled (reviewer check). The keys are surfaced, never
interpreted into a tree.
