# Spool check variant + ledger schema v2

**Before writing anything, read `docs/gotchas.md`** — the bugs prior slices
already bled on (entry guards, newline ordering, tests that cannot fail, the
degradation law). Reuse the fixes; do not re-derive them.

This is the campaign's persistence contract: the second spool-line variant
(the check line) and the complete ledger schema v2. Every later slice builds
against these shapes, which is why it lands first and why it is `high`. The
data design is settled — `docs/prd/PRD-0002-evidence-depth/schema.md` is the
authoritative source for every column and constraint below; on any conflict,
that file wins.

## The check line (second envelope variant)

The spool gains a second `v: 1` line shape, written by the CLI (never the
hook artifact — the hook artifact is byte-unchanged this campaign):

```
{ "v": 1, "ts": "<iso8601>",
  "check": { "name": "<str>", "argv": ["<cmd>", "<arg>", ...],
             "exit": <int>, "output": "<capped str>", "truncated": <bool>,
             "session_id": "<id>" | null,
             "bound_by": "single-open" | "explicit" | null } }
```

**Discrimination is by key, not by version.** A parsed `v: 1` line carries
exactly one of `event` (hook-event line, v1 path, unchanged) or `check`
(this variant). Neither, or both → corrupt: skip, count, name, occupy the
ordinal — a line that is neither is never guessed into either.

*Rejected shape, recorded:* bumping the envelope to `v: 2` for check lines.
`v` is the framing contract ("one JSON object per line, v + ts present,
readers skip what they don't recognize"); framing is unchanged. A per-variant
bump cannot work (hook lines stay `v: 1`), and a blanket bump would force
every existing `v: 1` reader to reject unchanged hook lines. A sibling
variant under `v: 1` is exactly the evolution the framing contract promised.

### Parse/serialize API — do not reshape parseEnvelope

PRD-0001 acceptance pins `parseEnvelope`'s behavior, including "a line with
no event member is a typed parse failure". **Preserve it.** Add a new
discriminating entry point (suggested name `parseSpoolLine`) returning a
typed union: event-variant (delegating to the existing envelope semantics,
byte-preserved payload and all), check-variant (the seven check fields,
exactly as written — `session_id` null and `bound_by` null pass through as
nulls, never defaulted), or a typed corrupt classification. Add a
check-serializer with the same guarantees the envelope serializer already
has: exactly one physical line, control-character-safe, round-trips
byte-identically, returns a typed failure on hostile input instead of
throwing. The trailing-newline rule from gotchas entry 2 applies.

## Ledger schema v2 (complete — later slices only populate)

Land the full v2 DDL in one stroke so no later issue alters schema. Deltas
from v1:

- `meta.schema_version` = **2**.
- `sessions` gains six nullable enrichment columns: `tokens_input`,
  `tokens_output`, `tokens_cache_read`, `tokens_cache_creation` (INTEGER),
  `cost_usd` (REAL), plus `model` and `cc_version` (TEXT). All NULL = ABSENT
  (degradation law) — tokens are stored separately from cost_usd so "tokens
  present, cost absent" is expressible and a re-ingest can retroactively
  regain cost.
- `events` gains nullable `background_task_id` (TEXT) — a promoted join key
  only; NULL on every non-backgrounding, non-TaskOutput event.
- New table `checks(line_no INTEGER PRIMARY KEY, ts TEXT NOT NULL, name TEXT
  NOT NULL, argv TEXT NOT NULL, exit_code INTEGER NOT NULL, output TEXT NOT
  NULL, truncated INTEGER NOT NULL CHECK (truncated IN (0,1)), session_id
  TEXT, bound_by TEXT CHECK (bound_by IN ('single-open','explicit')),
  CHECK ((session_id IS NULL) = (bound_by IS NULL)))`. `argv` is JSON array
  text verbatim; pass/fail is derived from `exit_code = 0` at render time,
  never stored as a boolean — the exit code is strictly richer (1 vs 137).
- New table `test_results(line_no INTEGER PRIMARY KEY, session_id TEXT NOT
  NULL, parser TEXT NOT NULL, passed INTEGER NOT NULL, failed INTEGER NOT
  NULL, skipped INTEGER NOT NULL, duration_ms INTEGER, failed_names TEXT NOT
  NULL)`. Row membership encodes the degradation law: no row = no parser
  claimed the command; a zero-count row = a real zero. `duration_ms` NULL =
  parser could not extract it, distinct from 0. `failed_names` is JSON array
  text, `'[]'` when failed = 0, never NULL.
- New table `absences(session_id TEXT NOT NULL, facet TEXT NOT NULL, reason
  TEXT NOT NULL, PRIMARY KEY (session_id, facet))` — at most one absence per
  facet per session.
- New indexes `idx_checks_session ON checks(session_id)` and
  `idx_test_results_session ON test_results(session_id)`. Nothing else — no
  index on `background_task_id` (the outcome join is an in-memory pass over
  one session's already-loaded events) and none on the cost columns.

**The schema_version bump is a rebuild trigger, never a migration.** Opening
a ledger whose `meta.schema_version` is not 2 deletes the database file and
recreates it fresh as v2 with `ingested_bytes = 0` and `lines_seen = 0`; the
next ingest rebuilds every projection from the spool. There is no ALTER path
and no migration tier, this campaign or any other. The ledger is disposable;
the spool (and transcripts-at-path) are the only ground truth.

## Invariants (contract prose, not acceptance)

- `parseEnvelope`'s v1 semantics are byte-for-byte preserved for event
  lines; every PRD-0001 envelope test stays green untouched.
- Check-line ordinals: a check line is a physical spool line and occupies a
  `line_no` exactly as event and corrupt lines do — identity is the ordinal,
  which is what makes rebuild-equivalence free downstream.
- No hard FK from `checks`/`test_results`/`absences` to `sessions` —
  `sessions` is itself derived (v1 ruling). A bound check may name a session
  whose events are not yet ingested; that is honest.
- The hook artifact knows nothing about any of this.

## Test-harness contract

Unit tests only — this contract has no seam-level behavior of its own.

- New check-variant parse/serialize tests live in the owned
  `tests/unit/core/checkLine.test.ts`; existing envelope tests are amended
  only additively.
- **One sanctioned amendment to a pinned PRD-0001 test:** the existing
  ledger unit test asserts `schema_version 1` on a fresh ledger. That
  assertion moves to 2 (and gains the v2 tables). This is the only
  pre-existing assertion this issue may change; the file is inside the
  declared footprint for exactly this edit.
- The rebuild-trigger test builds a real v1-shaped SQLite file (create the
  v1 tables and a meta row with schema_version 1 by hand in the test), opens
  it through the module, and asserts the file now holds an empty v2 schema
  with reset cursors.
- Export typed row readers for `checks`, `test_results` and `absences`
  from the owned ledger module (same read-API style as the existing
  session/event access) so later issues' acceptance tests import them
  directly and never re-implement SQLite access. The acceptance harness
  (`tests/acceptance/harness/`) is OUTSIDE this footprint — its generic
  readers stay byte-unchanged; this issue never edits it.
